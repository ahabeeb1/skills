## 0. Executive summary

Across 36 peer sources, every one of Modie's four pains has strong peer precedent for the proposed fix — but the cleanest implementation is *smaller* than expected, because three of the four problems collapse onto one converging primitive. The recommendation: ship **v1.22.0 as a five-piece bundle** (Plain-English plan format, Provisional-state HITL pivot, PostToolUse chain validator, Section-removal over tombstone for dormant dirs, and a markdown-only telemetry footer) plus a v1.23.0+ deferred-hardening ADR for plugin supply-chain. The headline trade-off: this *removes* surface (drop empty dirs, drop table-dominant plans, drop late-stage HITL gates) more than it adds — methodology shrinks rather than grows.

## 1. Problem (terse)

habeebs-skill v1.21.0 ships a research-grounded engineering chain that works, but four interlocking frictions remain: plans are unreadable (Jira-export aesthetic, zero peer precedent), hooks under-cover the chain (only 2 of 31 Anthropic events used, no chain-completion check), HITL gates fire too late (first review = `/grill`, after spec is drafted), and two ADR-declared methodology directories sit empty 6+ weeks past their carve-out. Critic added: mid-flight branches straddle methodology versions, plugin supply-chain has no Anthropic-blessed threat model, chain telemetry doesn't exist.

## 3. Patterns — converging, competing, and named for reference

### Eight converging patterns (named for synthesis reference)

1. **Queue + GC** (SP5, SP7) — accumulating per-event markdown files consumed and deleted at release time.
2. **Narrative-before-mechanics** (SP1, SP3) — 1-3 sentence prose intro opens every section.
3. **Artifact-existence-as-state** (SP2, SP5, SP7) — gate on file/directory presence; stateless under both Anthropic *and* ADR-0003's stricter bar.
4. **Provenance-scoped destructive ops** (SP5, SP6) — tools only touch paths they themselves created.
5. **Lifecycle-state metadata** (SP4, SP5, SP7) — frontmatter status enum + distinct `last-reviewed:` / `superseded-by:` fields, NOT single `date` overwrite.
6. **Cadence-driven editorial review** (SP4, SP5) — release-time human review handles dormancy; zero industry precedent for automated scanners.
7. **Early-gate before full artifact** (SP3, SP4) — approval BEFORE the cost of writing.
8. **Anthropic-guidance vacuum filled by community** (SP2, SP6) — Anthropic ships 31 hook events with no threat model; peers use 0-3 and ship security bundles to fill the gap.

### Six load-bearing competing patterns (synthesis picks one each)

| SP | Pattern A | Pattern B | Pick + reason |
|---|---|---|---|
| SP1 length | Loose budget | Tight budget | **A** — peers mandate *clear*, not *short*. Modie's pain is comprehension, not length. |
| SP2 enforcement | `PreToolUse` block | `PostToolUse` validator | **B for new chain-state checks, keep A for what already exists** — Pre is friction on every Edit/Write; Post composes with ADR-0003 statelessness naturally. |
| SP3 gate format | Yes/no + iteration | State-transition | **A** — 4/5 peers; matches Modie's stated need ("see and pivot"); state-transition adds complexity for solo-author cadence. |
| SP4 dormancy | Time-bound (ThoughtWorks) | No-time-bound (Rust) | **C — section-removal over tombstone (MADR)** — don't declare empty scaffolding in the first place. Cleanest fit for ADR-0002. |
| SP5 in-flight | Manual reclassification gate (Backstage) | Three-stage deprecation (superpowers) | **A** — solo-author cadence makes Backstage's blocking gate cheap; superpowers' 2-minor patience is overkill at Modie's release pace. |
| SP6 threat model | Acknowledge-gap-with-framework | Ship hooks-as-mitigation | **A** — only one that doesn't contradict ADR-0002; honest, defensible, deferable. |

### Six silent contradictions surfaced + resolved

1. **"Early gate" vs `provisional` state vs design sign-off (SP3)** — Resolved: pick KEP `provisional` shape (stub exists, status pending) over PEP pre-RFC issue gate (no document yet). Matches habeebs-skill's existing artifact lifecycle.
2. **Tombstone semantics (SP4)** — Resolved: tombstone applies to records WITH content (Nygard/Fowler); section-removal applies to empty scaffolding (MADR). habeebs-skill's ADR-0018 conflated the two; v1.22.0 separates them.
3. **"Stateless" hooks (SP2)** — Resolved: habeebs-skill's ADR-0003 is stricter than Anthropic's "no in-memory state" definition. Disk-existence checks ARE stateless under both bars; session-state directories are NOT.
4. **No time-bound retirement (SP4) vs drop policy (SP7)** — Resolved: Rust rejects time bounds; ThoughtWorks uses them. Pick ThoughtWorks for dormant *directories* (Piece 4), keep Rust-style for *ADR records with content* (existing ADR-0021 Pattern B).
5. **"Drop tables" vs "fix jargon" (SP1)** — Resolved: BOTH must ship together in Piece 1. Dropping tables alone solves half the problem.
6. **Provenance-scoped reaping (SP5) vs wide hook bundles (SP6)** — Resolved: pick narrow (superpowers' `.worktrees/`-only scope). habeebs-skill's PostToolUse validator scopes to plugin-owned paths only.

## 4. Recommendation for habeebs-skill v1.22.0

**Ship a 5-piece bundle that *removes* methodology surface rather than adds it, plus 1 deferred-hardening ADR.**

### Piece 1 — Plain-English plan format (SP1)

Rewrite `write-plan` SKILL.md and `references/plan-template.md` to enforce:

- **Top of doc:** TL;DR section (3-5 plain-English sentences) before any table or status block. Reader picks up the plan cold and knows what's happening.
- **Per-phase narrative intro:** Every phase has 1-3 sentences of "why this phase exists / what we accomplish here" BEFORE any acceptance gate, risk list, or table.
- **Tables collapse to TWO uses only:** (a) Status block at top (Plan ID / ADR / Tier / Status / Owner — 5 rows max), (b) Slice list. Acceptance gates become numbered prose lists. Risks become prose paragraphs with embedded "Mitigation:" lines, not nested bullets.
- **Jargon glossary:** First use of `pgroup`, `AFK:full-auto`, `HITL:approval-gate`, `pre-flight verification` etc. is inline-defined OR linked to a glossary section at bottom of plan. Three uses or fewer in body → inline. More than three → glossary entry.
- **Drop:** rollback-hook columns, blocked-by tables that read like Jira, 3-level-nested risk hierarchies.

Pattern-extractor's anti-smoothing flag: "drop tables" alone solves half the problem; **the other half is jargon discipline** — both must ship together.

### Piece 2 — Provisional-state HITL pivot (SP3, addresses Modie's stated pain directly)

Insert a new HITL gate between `prior-art-research` Phase 6 (synthesize) and Phase 6.5 (archive). The chain becomes:

```
/research → Phase 6 synthesize → [NEW: pivot review] → Phase 6.5 archive → /spec
```

**Mechanics:**

- Phase 6 output ends with `## Phase 6 ready for HITL review — pivot point` block.
- User responds with one of: **(a) approve as-is**, **(b) approve with pivots** (free-text edits), **(c) reject + re-research with new scope**.
- Free-text response gets applied as plain edits to the in-conversation report BEFORE Phase 6.5 commits the archive. Once Phase 6.5 writes the file, it IS the source of truth — downstream skills read the archive, not the conversation.

We're picking the **KEP `provisional` shape** (stub document exists, status pending) over PEP's **pre-RFC issue gate** (no document yet). The report IS the document; the pivot is on its content, not on whether to write it. This matches habeebs-skill's existing artifact lifecycle.

**Why this and not "approve before /spec":** Same pivot point, lower latency. By the time the user sees the recommendation, they have everything they need to pivot. Adding a separate `/spec`-pre-gate would double-gate the same decision.

### Piece 3 — PostToolUse chain-state validator hook (SP2, SP5)

Add ONE new hook to `hooks/hooks.json`: `PostToolUse[Edit|Write|NotebookEdit]` running a stateless `check-chain-state.sh` validator. The validator:

- Reads the current working directory + repo state (file existence only — ADR-0003-compliant)
- Detects "starting a new feature": new commit on non-default branch + first content change to `skills/` or `hooks/` or `.claude-plugin/`
- Checks chain-artifact existence per the scope picks made in `/grill` (see Open question § Piece 3 scope)
- **Warns** (never blocks) when expectations don't hold. Exit code 0 always; warning text printed to stderr. Modie reads the warning and decides.

This is the "artifact-existence-as-state" pattern. It does NOT track "are we in /tdd right now" (that would be state-owning, ADR-0003 violation). It only checks "do the artifacts that *should* exist if we're doing X actually exist?"

**Drop or defer:** the `Stop` / `SubagentStop` hook idea from the brief. Closest peer precedent (disler) uses `SubagentStop` for TTS announcements only, not chain verification. Validation belongs to `PostToolUse` per silent-contradiction-3.

### Piece 4 — Section-removal over tombstone for dormant dirs (SP4)

Delete `docs/agents/dispatches/` and `docs/agents/conflicts/`. Amend ADR-0021 with a `## 2026-05-26 Re-Amendment` block:

> The 2026-05-25 carve-out (preserve `dispatches/` + `conflicts/` as future-use audit-log directories) is REVERSED. Six weeks post-carve-out, both directories remain empty. ThoughtWorks Tech Radar's "one edition without movement → fade" threshold is cleared 1.5×. MADR v4's "remove the empty optional section" principle applies: declared-but-empty containers are not tombstones — they're scaffolding that didn't earn its existence. ADR-0004 Part 2 and ADR-0018 Part A keep their Status: Accepted; their declared directories are deleted; future re-introduction requires either an actual writer landing the first file in the same PR, OR a fresh ADR with population evidence.

**Future rule (codified in ADR-0021 re-amendment):** Any future ADR declaring a new methodology directory MUST land its first file in the SAME PR. No empty scaffolding shipped.

### Piece 5 — Markdown-only chain telemetry (SP7)

Add three fields to plan + ADR frontmatter:

```yaml
---
status: <Proposed | Accepted | Active | Done | Deprecated | Superseded by ADR-N>
date-created: 2026-05-26
last-reviewed: 2026-05-26  # distinct from date-created; updated on transition
superseded-by: null         # or path to replacement ADR/plan
---
```

Drop the single `date:` overwrite pattern (MADR's gap). Add a release-time editorial step in `release` skill Phase 10:

- Walk `docs/agents/adrs/*.md` + `docs/agents/plans/*.md`
- Flag any with `status: Proposed` or `status: Active` and `last-reviewed` more than 3 releases ago
- Print the list as a warning; don't block release. Modie reads, decides, optionally lands an update PR (today's chore PR #47 pattern).

This is the **Cadence-driven editorial review** pattern (SP4 + SP5 convergence). No automation, no scanner — just a release-time grep.

### Piece 6 — Deferred-hardening ADR for plugin supply-chain (SP6)

Write a new ADR (`adr-plugin-supply-chain-threat-model.md`) that:

- Names the gap (Anthropic ships no threat model)
- Cites OWASP CICD-SEC-8 as upstream framing
- Cites Shai-Hulud as evidence the threat is actual
- Inherits ADR-0003 (stateless hooks) + ADR-0004 Part 5 (untrusted-content rule) as existing partial defenses
- Defers hardening (signature verification, hook sandboxing, file-content-as-data marker) to v1.23.0+ with named research targets
- Records `Status: Accepted` (the ADR is real; the deferral is the decision)

This is the **Acknowledge-gap-with-framework** pattern. The honest deliverable for v1.22.0 is the named gap, not invented hardening.
