# Prior-Art Research — Dated artifact naming vs. release-driven late-binding integers

**Tier:** Deep — 5 sub-problems, reopens a documented ADR rejection (ADR-0020), machinery-teardown + 1,241-cross-ref-integrity constraints.
**Date:** 2026-05-28
**Status:** Archived (HITL pivot: approved as-is 2026-05-28 → `/spec`)
**Feature slug:** v1.23.0-dated-artifact-naming
**Researcher:** prior-art-research (Deep tier, 5 source-fetchers + 1 category-completeness critic)

## 0. Executive summary

The user's instinct is **right about the problem, half-right about the precedent, and the cleanest fix isn't the one they named.** The real pain — ADR identity coupled to the release event ("not everyone has releases") — is legitimate, and ADR-0020's claim of "zero canonical endorsement for dated naming" is **overstated** (log4brains defaults to it). But "dated like Superpowers/Kiro" doesn't hold up: Superpowers dates **plans/specs and keeps no ADRs at all**, and Kiro **dates nothing** (slug dirs). Recommendation: **write ADRs with their final identifier at creation time** (kill the release-rename), and adopt `YYYY-MM-DD-<slug>.md` for ADRs, specs, plans, and grill-records — with **the slug as the real uniqueness key** (date sorts, slug disambiguates same-day), exactly as log4brains and Shopware ship it.

## 1. Problem

ADR identity is assigned by the `release` skill (decision-record writes `adr-<slug>.md`; release renames to `NNNN-<slug>.md` at release time, per ADR-0020). A decision therefore has **no stable identifier until a release happens** — wrong for a methodology meant to work in repos with no release cadence. Evaluate: move to dated naming, keep late-binding integers, or hybrid.

## 2. Case studies

- **obra/superpowers** ([github.com/obra/superpowers](https://github.com/obra/superpowers)) — Dates **plans and specs**: `writing-plans/SKILL.md` mandates `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`. Keeps **zero ADRs**. Confirms the premise for plans/specs; not a dated-ADR precedent.
- **AWS Kiro** ([kiro.dev/docs/specs](https://kiro.dev/docs/specs/)) — Per-feature **slug dir** (`.kiro/specs/<slug>/{requirements,design,tasks}.md`). **No dates, no ADRs.** Counter-example to "dated like Kiro."
- **github/spec-kit** ([github.com/github/spec-kit](https://github.com/github/spec-kit)) — Sequential `001-feature/` dirs; dates only behind opt-in `--timestamp` (`YYYYMMDD-HHMMSS`, not `YYYY-MM-DD`). Same family as the current scheme.
- **log4brains** ([github.com/thomvaill/log4brains](https://github.com/thomvaill/log4brains)) — Mainstream ADR tool that **defaults to `YYYYMMDD-title.md`**, slug as canonical ID, date "only used for rough sorting." Migrated *from* integers because they collide on `git merge`. The dated-ADR precedent ADR-0020 missed.
- **Shopware ADRs** ([developer.shopware.com](https://developer.shopware.com/docs/resources/references/adr/)) — Ships `YYYY-MM-DD-title.md` in production 2020→2026, release-independent. Proof dated ADRs run fine at product scale.
- **Rust RFCs** ([github.com/rust-lang/rfcs](https://github.com/rust-lang/rfcs)) — Draft `0000-foo.md`; number bound **at merge = PR number**. The cleanest decouple: release-independent, collision-free, monotonic — but needs a forge to supply the number.
- **Rails migrations** ([guides.rubyonrails.org](https://guides.rubyonrails.org/active_record_migrations.html)) — Switched sequential→`YYYYMMDDHHMMSS` in 2.1, **left old files untouched** in the same dir. Shipped precedent for "freeze 0001-0024, date new ones."

## 3. Patterns

| Pattern | Who | Decouples from release? | Same-day collision-safe? | Causal ordering | Fit |
|---|---|---|---|---|---|
| Sequential int, bound at release | current (ADR-0020) | ❌ no | ✅ (late-bind) | ✅ strong | the pain being fixed |
| Sequential int, bound at **merge** (PR#) | Rust, Backstage | ✅ yes | ✅ | ✅ strong | needs forge number; no CI here |
| `YYYY-MM-DD-<slug>` (date + slug-as-key) | log4brains, Shopware, Superpowers(plans) | ✅ yes | ✅ *via slug* | ⚠️ weaker | **best fit** |
| Slug-only, date in frontmatter | joelparkerhenderson | ✅ yes | ✅ via slug | ⚠️ weakest (no sort) | loses sortability |

Three load-bearing findings:

1. **Bare `YYYY-MM-DD` is NOT collision-proof.** Two ADRs the same day collide. Every shipped dated system resolves this with **the slug as the real unique key** (log4brains, Shopware) or **sub-day timestamps** (Rails `HHMMSS`). The unique identifier is `date + slug`, not the date alone — material because the user runs parallel Claude Code sessions (the exact collision class ADR-0020 killed). The MADR canonical thread ([adr/madr#28](https://github.com/adr/madr/issues/28)) states verbatim that with dates "merge conflicts can still occur if two ADRs are created at the same time."
2. **Causal ordering degrades.** ADRs cross-reference ("superseded by ADR-0006"). A monotonic integer says "came after" for free; a date is fuzzy (same-day = no order) and a slug says nothing. Supersession must lean entirely on explicit forward-links.
3. **dogfood-28 is not violated.** Scenario 28 bans dated *strings in prose* (version-archaeology that rots). A dated *filename* is an immutable identifier asserting "decided on D" — permanently true, like a git SHA. Cleanly different concern; needs only a one-line lint carve-out for ADR/spec/plan filenames + frontmatter.

## 4. Recommendation for this context

**(A) Decouple identity from release — the real fix.** `decision-record` writes the **final filename at creation**, not `adr-<slug>.md` awaiting a release rename. This is the change that addresses "not everyone has releases," independent of date-vs-integer.

**(B) Use `YYYY-MM-DD-<slug>.md` for new ADRs, specs, plans, grill-records.** Date prefix sorts chronologically and is legible; **the slug is the collision-disambiguator** (never the date alone). Matches Superpowers for plans/specs, log4brains/Shopware for ADRs, and what `docs/agents/research/` already does. Reference new ADRs by `date-slug`; keep `ADR-00NN` for the 24 legacy ones.

**Migration: freeze-old / date-new (the Phase 1 choice; correct).** ADRs 0001-0024 keep integer names — zero broken cross-refs across ~117 files / 1,241 references. New ADRs dated. Mixed directory is shipped precedent (Rails 2.1), not a smell. Signal the cutover with a marker note in `adrs/README.md` (Cloud Posse's dedicated-section pattern), since the repo is read directly by humans and agents without sorting tooling.

**Push-back recorded:** if collision-safety and clean supersession ordering were paramount, the Rust "number-at-merge = PR number" pattern is objectively cleaner than dates — but it requires reading the PR number at merge, and this project has no CI and merges manually. Dates+slug is the better fit for a no-CI, markdown-only, parallel-session solo project; the slug carries the meaning and supersession is already expressed by explicit forward-links, not numeric adjacency.

## 6. Specific decisions to make next (→ `/grill`)

1. **ADR same-day uniqueness rule:** confirm `date + slug` is the key (slug disambiguates) and `decision-record` halts loud on same-date-same-slug — OR add a `-2` sequence suffix. Which?
2. **Cross-reference style going forward:** new ADRs referenced as `[[2026-05-28-foo]]`? How do new ADRs cite *old* ones (still "ADR-0007")? Lock the bidirectional convention.
3. **Specs/plans rename trigger:** currently `vX.Y.Z-<slug>.md`. Switch new ones to `YYYY-MM-DD-<slug>.md` — does spec→plan→release traceability (which keys off the version) survive losing the version in the filename? Version moves to frontmatter?
4. **Cutover marker:** README note vs. tombstone ADR vs. frontmatter `scheme:` field.
5. **README index regeneration:** the index table is integer-keyed and was auto-appended by the (deleted) script. New tiny script, or hand-maintained for dated ADRs?
6. **ADR-0020 disposition:** partial-supersession amendment vs. full new superseding ADR. Changesets half must survive.

## 7. Open questions

- **Causal ordering loss** is the one genuine regression. Is supersession-by-forward-link strong enough that "ADR-7 came before ADR-12" is never missed?
- **No source on multi-agent legibility** of a mixed integer+dated dir (SP4 tier was dry here). Low risk, unverified.
