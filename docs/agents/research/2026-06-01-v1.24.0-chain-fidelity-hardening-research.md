# Prior-Art Research: habeebs-skill self-enhancement (v1.24.0 chain-friction hardening)

**Researched on:** 2026-06-01
**Tier:** Balanced
**Sources consulted:** 8

## TL;DR

Of the six chain-friction candidates, **three warrant v1.24.0 and three should be cut or downgraded**. Ship the cluster that hardens existing gates with executable assertions: **(SP2) a migration-archaeology prose lint** extending dogfood 26/27/28, **(SP3) a "confirm fixture IDs against the live tree" rule** in draft-spec/write-plan/tdd-loop, and **(SP4) a doc-sync supersession-link integrity check**. Each is markdown/bash-only, extends an ADR you already have rather than minting a new architecture, and carries a clean dogfood test. The single biggest trade-off: every new lint widens the false-positive surface, so each must ship with a `pattern-not`-style carve-out and be corpus-tested before it blocks — the same discipline ADR-0022 already established. The cwd-blind-hook lesson (SP1) is **already fixed in v1.23.0**; generalize it as a one-paragraph ADR-0003 amendment, not a build slice. The runtime-lag doc (#1) is a README footnote, not a release.

## Context

- **Building:** Self-enhancement of the habeebs-skill plugin — hardening the chain against six friction modes observed running it end-to-end during v1.23.0.
- **Scale:** Public OSS Claude Code plugin; install count [unknown — untracked]. 19 skills, 25 ADRs, 36 dogfood scenarios, 6 hooks.
- **Stack:** Markdown SKILL.md instruction files + bash dogfood tests + bash hooks. No build system, no runtime.
- **Constraints:** ADR-0002 — no runtime substrate (no MCP, daemons, vector stores, session-state dirs). Every fix must be markdown or bash.
- **Existing:** Brownfield, methodology-mature (v1.23.0 current on `main`).
- **Priorities:** Shipping speed + operational simplicity (one tight v1.24.0 release).

## Sub-problems

1. **SP1** — Hooks/guards must resolve their predicate from the action's **target**, not the hook's process context (candidates #2, #4).
2. **SP2** — Lint behavioral docs: ban migration/transition/archaeology prose in SKILL.md bodies (#5).
3. **SP3** — Specs/plans must not hard-code identifiers that only bind at implementation time (#3).
4. **SP4** — Release/doc-sync must verify referential integrity of superseding decision records (#6, plus #1 as a doc-only sibling).
5. **SP5** — Gate hardening: any new blocking assertion needs false-positive blast-radius analysis, a documented override/escape, and an overlap map against the live 36-scenario suite (Phase 2.5 critic additions).

## Case studies

### Anthropic Claude Code — WorktreeCreate hook `worktree_path` vs `cwd` (issue #36556)

- **Architecture:** The official WorktreeCreate hook prints a single path used for both git-cleanup matching and session-cwd selection.
- **Key decision:** A user requested splitting the output into two fields because in a monorepo the worktree root (cleanup target) and the session dir (subfolder) differ; conflating them forces "correct cwd OR functional cleanup, not both."
- **Scale:** Anthropic's first-party hook system — the canonical reference for what "well-designed hook context" looks like.
- **Trade-off accepted:** Closed as not-planned — so the ecosystem's own first-party hooks still carry this conflation. The *lesson* is validated even though the upstream fix isn't.
- **Source:** [anthropics/claude-code#36556](https://github.com/anthropics/claude-code/issues/36556) — "Single output path forces choice between correct CWD or functional cleanup."

### pre-commit & mise — worktree hook-directory resolution footgun

- **Architecture:** Both tools install/generate git hooks; both had bugs where a hook resolved paths from the wrong location under a worktree.
- **Key decision:** The fix in both cases is to *dereference the real gitdir* rather than assume the launch directory — i.e., resolve from the action's true target, not the ambient cwd.
- **Scale:** pre-commit is the dominant Python hook framework; the bug recurs independently in mise.
- **Trade-off accepted:** Worktree support adds path-resolution complexity every hook author must handle.
- **Source:** [pre-commit#808](https://github.com/pre-commit/pre-commit/issues/808), [mise#4853](https://github.com/jdx/mise/discussions/4853)

### Vale + GitLab docs — tense/time linting for "living" docs

- **Architecture:** Vale is a prose linter; teams write regex/substitution rules that flag time-bound language ("will be" → present tense) to keep docs version-agnostic.
- **Key decision:** GitLab's Vale ruleset bans future tense outright; the broader guidance is "focus on the current version to reduce maintenance."
- **Scale:** Vale is the de-facto docs-as-code prose linter (Elastic, Grafana, GitLab, Microsoft/Google style imports).
- **Trade-off accepted:** Prose linting produces false positives; teams accept a carve-out/allowlist burden.
- **Source:** [Vale](https://vale.sh/docs), [GitLab Vale rules](https://docs.gitlab.com/development/documentation/testing/vale/)

### Semgrep / Conftest / commitlint — policy-as-code rollout

- **Architecture:** Convention encoded as an executable rule that runs on every PR and returns a nonzero exit to block.
- **Key decision:** Roll out by (a) `pattern-not` clauses to filter false positives, (b) testing the rule against the existing corpus *before* enabling, (c) choosing block-vs-alert per rule.
- **Scale:** Industry-standard; commitlint+Husky is the canonical commit-convention enforcer.
- **Trade-off accepted:** Every rule is itself a maintained artifact (a "rule catalog" that needs dedup and review).
- **Source:** [Semgrep methodology](https://semgrep.dev/blog/2020/writing-semgrep-rules-a-methodology/), [commitlint](https://github.com/conventionalcommit/commitlint)

### log4brains / adr-tools — supersession is link-but-don't-validate

- **Architecture:** ADR tooling models statuses (proposed/accepted/deprecated/superseded) and the convention is "write a new record + link the two together."
- **Key decision:** Forward-linking on supersession is universal; **automated referential-integrity validation of those links is not a documented feature** of either tool.
- **Scale:** log4brains and adr-tools are the two most-cited ADR managers.
- **Trade-off accepted:** Link integrity is left to author discipline — exactly the gap that bit v1.23.0's partial supersession of ADR-0020.
- **Source:** [log4brains](https://github.com/thomvaill/log4brains), [adr-tools](https://github.com/npryce/adr-tools)

## Patterns

### Pattern A — Resolve-from-target, not from ambient context

A guard/hook must derive its decision input from the *thing being acted on*, never from the process's incidental launch context. The commit-block hook read branch from hook-process cwd (`main`) instead of the worktree being committed to; Claude Code's own WorktreeCreate hook conflates `worktree_path` with `cwd`; pre-commit/mise resolved from the wrong gitdir. Same bug, four codebases.

**Fits when:** any block-only or context-injecting hook whose correctness depends on *where* or *what* the action targets — i.e., all of them.

### Pattern B — Convention-as-executable-assertion, rolled out corpus-first

Encode a methodology rule as a test/lint that runs on every change, but de-risk it before it blocks: write `pattern-not`-style carve-outs, run it against the whole existing corpus to measure the false-positive count, and only then make it gating. This is exactly how habeebs-skill's dogfood 26/27/28 already work (bash regex over bodies, with HTML-comment exclusions), and it matches semgrep/Vale/commitlint practice.

**Fits when:** adding SP2 (prose lint) and SP3 (fixture-ID check). The corpus-first step is non-negotiable — it *is* the SP5 mitigation.

### Pattern C — Link-and-validate (the ecosystem only does link)

Supersession links are a universal convention, but no surveyed ADR tool *validates* them. A doc-sync check that asserts "status=Superseded ⟹ forward link exists, and for partial supersession both records name the surviving half" is a small, ecosystem-novel hardening — low cost, real catch.

**Fits when:** SP4. The novelty is mild and the surface is tiny (it runs only when an ADR's status flips), so the risk is low.

## Recommendation

**For a one-tight-release v1.24.0, ship SP2 + SP3 + SP4 as a single "chain-fidelity" bundle, generalize SP1 as a one-paragraph ADR-0003 amendment (no code — already fixed), and demote #1 to a README line.**

The three build items (SP2/SP3/SP4) share a single shape — *executable assertions that catch a class of error a human had to catch manually this session* — and all three extend existing ADRs (0022, the spec/plan conventions, and the release doc-sync audit) rather than minting new architecture. That keeps the changeset coherent and the release tight. SP1 is the highest-*pain* item but its code already shipped in v1.23.0 (`349bba4`); the residual work is purely doctrinal — amend ADR-0003 to state the resolve-from-target rule as a fourth hook design rule so the next hook author doesn't reintroduce it. SP5 is not a slice; it's the *acceptance discipline* applied to SP2 and SP3 (corpus-test before blocking; document the carve-out). #4's subagent-commit-path problem is now moot — the same v1.23.0 hook fix that resolved SP1 means `cd <worktree> && git commit` works for dispatched subagents too; capture it as a one-line note in parallel-dev's commit-discipline section, not a slice.

Given the "shipping speed + operational simplicity" priorities, I explicitly reject building a general prose-linter integration (Vale) — it's a runtime tool ADR-0002 forbids, and bash regex in the dogfood harness already does the job at zero new dependency.

### What you're explicitly giving up

- **No general prose linter (Vale/CI integration).** Bash regex in dogfood covers it; a real linter would violate ADR-0002 and add a dependency.
- **No upstream hook fix for the WorktreeCreate `worktree_path`/`cwd` conflation** — that's Anthropic's, and they closed it not-planned. We only fix our own hook (done) and document the lesson.
- **SP5 ships as discipline, not as a feature** — there's no "gate-hardening framework," just the rule that SP2/SP3 must be corpus-tested before blocking.

### When to revisit

- If dogfood 34 (SP2 prose lint) produces >1 false positive per release cycle → re-evaluate the regex or codify a new carve-out (mirrors ADR-0022's existing revisit trigger).
- If a third "resolve-from-target" hook bug appears → promote Rule 4 from an ADR-0003 amendment to a standalone hook-authoring reference doc.
- If the dogfood suite crosses ~50 scenarios → the SP5 "suite-interaction map" concern becomes a real maintainability project (rule-catalog dedup), worth its own research.

## Decisions to make next

These feed `socratic-grill` and `draft-spec`:

1. **Bundle boundary** — Is v1.24.0 exactly SP2+SP3+SP4+the SP1 amendment, or trim to the single highest-leverage item (likely SP2)? The "one tight release" answer leans to all-four-but-small; grill should pressure-test whether SP3 and SP4 each clear the bar.
2. **SP2 banned-phrase precision** — The exact regex set. Too broad false-positives on legitimate present-tense prose ("X replaces Y at runtime"); too narrow misses archaeology. Must be corpus-tested against all 19 SKILL.md bodies before locking — what's the acceptable false-positive count to ship?
3. **SP2 warn vs block** — Does scenario 34 fail CI (block, like 26/27/28) or warn first? ADR-0003's warn-only/block-only dichotomy plus Semgrep's "test corpus first" suggest: block, but only after the corpus test shows zero current offenders.
4. **SP4 scope** — Does the supersession check live in the release skill's doc-sync (runs at release) or as a standalone dogfood scenario (runs every PR), or both? Both is cheap; grill should confirm.
5. **SP1 amendment shape** — Does Rule 4 amend ADR-0003 in place (like the ADR-0015 tag-push amendment did) or get its own dated ADR? In-place amendment matches precedent and keeps it tight.
6. **Does any of this warrant an ADR at all?** SP2/SP3 are convention refinements; SP4 is a new (if small) invariant. Likely one dated ADR covering the bundle + the SP1 in-place amendment to ADR-0003. Grill/record decides.

## Open questions

Things research didn't resolve:

- **Corpus false-positive rate for SP2 is unmeasured.** I have not yet run the candidate regex against the 19 live SKILL.md bodies — that measurement is the first implementation step and could reshape the banned-phrase set materially.
- **Whether the two known-red scenarios (16 session-summary-template, 29 chain-state-validator) interact with the new scenarios.** They're pre-existing reds on `main`; SP5's suite-interaction concern says to check, but they're likely orthogonal. Triaging them is arguably its own tiny cleanup, out of this bundle's scope.
- **Whether #1 (runtime-lag) is worth even a README line**, or whether it's so obvious to anyone who's developed a self-installing plugin that documenting it is noise. Leaning "one line in tdd-loop Phase 0," but a grill could cut it entirely.
