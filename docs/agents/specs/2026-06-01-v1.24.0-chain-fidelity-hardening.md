## TL;DR

Harden the chain against three friction modes that bit during v1.23.0, plus lock one already-shipped fix as doctrine. We add three executable assertions — a migration-archaeology prose lint (extends dogfood 26/27/28), a "confirm fixture IDs against the live tree" rule in the spec/plan/tdd surfaces, and a supersession-link integrity check in release doc-sync — each markdown/bash-only, each extending an existing ADR. Five slices, all small; SP1's code already shipped in v1.23.0 so its slice is a one-paragraph ADR-0003 amendment, not new code.

## Architecture

Three independent executable-assertion additions plus two doctrine edits, all in markdown + bash. No new dependency, no runtime substrate (ADR-0002). Each lint follows the existing dogfood idiom: a bash script with an awk body-extractor (frontmatter/HTML-comment/footer/code-fence excluded), per-case `mktemp -d` fixtures, a single-source-of-truth guidance string, a final `(e) main repo scan` that must pass on the live corpus, and a `===SCENARIO N ALL M CASES PASS===` banner. The "corpus-first" rollout (SP5 discipline) means each new lint is run against the live tree and its false-positive count driven to zero *before* it is allowed to block.

```
SP2 prose lint        → tests/dogfood/34-no-migration-archaeology/   (new, mirrors 28)
SP3 fixture-ID rule   → skills/draft-spec + skills/write-plan + skills/tdd-loop  (instruction text)
                      → tests/dogfood/35-fixture-id-late-binding/     (new, asserts the instruction present)
SP4 supersession check→ skills/release doc-sync + tests/dogfood/36-supersession-link-integrity/  (new)
SP1 doctrine          → docs/agents/adrs/0003-hooks-scope.md          (in-place amendment, Rule 4)
#1 runtime-lag        → README.md Hooks section + skills/tdd-loop Phase 0  (one line each)
```

## Trade-offs accepted

- **No general prose linter (Vale/CI integration).** Bash regex in dogfood covers it; a real linter would violate ADR-0002 and add a dependency.
- **No upstream hook fix for the WorktreeCreate `worktree_path`/`cwd` conflation** — that's Anthropic's, and they closed it not-planned. We fix only our own hook (done) and document the lesson.
- **SP5 ships as discipline, not a feature** — no "gate-hardening framework," just the rule that SP2/SP3 lints are corpus-tested before they block.
- **Each new lint widens the false-positive surface.** Mitigated by the corpus-first rollout + carve-out mechanism; the cost is real and accepted.

### Slice 1 — SP2 migration-archaeology prose lint (HITL)

**Description:** Add dogfood scenario `34-no-migration-archaeology` that scans every `skills/*/SKILL.md` body for **self-referential archaeology prose** — sentences narrating the methodology's own evolution — and fails (blocks) if found. Mirror scenario 28's structure exactly. The grill already ran the corpus test: the naive phrase set scored 4 hits / 100% false-positive, so the regex is the **narrowed subject-aware set** (below), which corpus-tests to zero on the v1.23.0-cleaned tree. HITL because the regex precision needs human ruling on edge cases as new banned shapes are considered.

**Acceptance criteria:**
- [ ] `tests/dogfood/34-no-migration-archaeology/check-no-archaeology.sh` exists, follows the scenario-28 idiom (awk body-extractor: frontmatter + HTML-comment + footer + code-fence excluded; `mktemp -d` fixtures; SSOT guidance string; final banner)
- [ ] Regex targets self-referential archaeology: `this skill used to`, `previously (this|we)`, `was renamed`, `replaces the old <X>`, `(in|set in) vN this changed`, `formerly`, `no release step <verb>s it`, `<X> are NOT renamed` — NOT bare `no longer`/`used to`
- [ ] **Positive cases plant the session's ACTUAL reintroduced offenders** ("this skill used to write adr-<slug>.md", "Existing X are NOT renamed (freeze-old)", "no release step renames it") — proving the regex catches the real failure mode, not a strawman
- [ ] Negative cases prove no false positive on the 4 measured FP shapes: "Deprecated — no longer current", "a test that used to pass now fails", "no longer maps to reality", "a path that no longer exists" — all NOT flagged
- [ ] Negative cases also prove the carve-out: frontmatter, HTML-comments, footer, code fences NOT scanned
- [ ] The `(e) main repo scan` case PASSES — narrowed regex hits zero on the live corpus (the v1.23.0 cleanup already removed the real offenders)
- [ ] Lint is BLOCKING (exit nonzero on hit) — joins the regression suite like 26/27/28. (Warn fallback documented as non-triggered contingency only.)
- [ ] Failure message (SSOT guidance string) tells a tripped contributor how to fix (restate in present tense; archaeology belongs in CHANGELOG/git history/postmortems)
- [ ] Banner `===SCENARIO 34 ALL N CASES PASS===` emitted

**Test strategy:** The slice IS a test (bash dogfood). RED = the new script firing on the planted-offender fixtures (and, if any real offenders remain, on the live corpus); GREEN = all positive cases caught + all 4 measured-FP shapes pass + live corpus clean — at `tests/dogfood/34-no-migration-archaeology/check-no-archaeology.sh`

**Blocked by:** None

**Notes:** Confirm `34` is still the next free number at implementation (SP3 self-application — confirmed free at grill time, highest live is 33). Watch it fire on the planted offenders first (tracer), then confirm it's silent on the clean corpus.

## Parallelization

AFK slices with no shared dependencies can run via `parallel-dev`:

- **Group A (parallel, AFK):** #2 (SP3 fixture-ID rule), #3 (SP4 supersession check) — disjoint write scopes (`skills/draft-spec`+`write-plan`+`tdd-loop`+`tests/dogfood/35` vs. `skills/release`+`tests/dogfood/36`). Independence to be re-verified at parallel-dev Phase 2.
- **Sequential:** #1 (HITL — gates the corpus regex), then #4 (HITL — depends on #1), then #5 (HITL — release).

Note: #1 is HITL and #4 depends on it, so the realistic order is: #1 first (HITL), Group A (#2+#3 parallel AFK) can run concurrently with or after #1, then #4 (HITL) once #1 lands, then #5.

**No self-application ordering coupling (grill Item 8).** SP3 (Slice #2) adds the "confirm fixture IDs against the live tree" rule, and Slices #1/#3 themselves create dogfood scenarios (34/36). This is NOT a dependency: SP3 is doctrine the implementer follows, not a runtime gate that must exist first. #1 and #3 confirm their scenario numbers at creation regardless of whether #2's written rule has landed (demonstrated at spec time — 34 confirmed free against the live tree). `parallel-dev` Phase 2 should NOT re-flag this as a coupling.
