## Per-item resolutions

### Item Q1 — Postmortem implementation shape

- **Starting state:** Spec listed both skill and section paths; HITL gate on Slice 3.
- **Axes grilled:** Reversibility (section → skill is non-breaking; skill → section IS breaking), Observability (discoverability via description-driven trigger), Migration (description-budget pressure from v1.9.0 trim).
- **Key questions asked:** Is the umbrella `using-habeebs-skill` description sufficient surface for postmortem trigger? Would a separate skill regress the v1.9.0 description budget?
- **Resolution:** DECIDED
- **Decision:** **Section in `using-habeebs-skill` for v1.10.0; promote to skill in v1.11.0 if uptake is high.** Revisit trigger: 10+ real postmortems land in 90 days OR users explicitly request a dedicated `/postmortem` skill.
- **Spec update:** Slice 3 drops HITL label (the gate is closed) and the if-Q1=skill branch. ADR-0011 frames postmortems as a doc convention reachable from the umbrella skill, with explicit "v1.11.0 promotion criterion" section.

### Item Q2 — R1 release shape

- **Starting state:** Spec offered v1.10.0 patch vs v2.0.0 breaking change.
- **Axes grilled:** Migration (auto-migrate via Phase 0 single-writer; no user action), Reversibility (additive-by-deletion — chain skills tolerate missing sections), Failure modes (what breaks if a downstream skill reads a now-dropped field).
- **Key questions asked:** Does any chain skill currently read Stack / Persistence / Deployment / External services / Recent hot files as load-bearing? Answer from grep audit: no — these sections are inferential context only.
- **Resolution:** DECIDED
- **Decision:** **Patch release v1.10.0 with Phase-0-next-write auto-migration.** No migration skill needed.
- **Spec update:** None — Slice 1 already reflects this path. Lock the version-bump in Slice 6 at 1.9.0 → 1.10.0.

### Item Q3 — verify-output / postmortem relationship

- **Starting state:** ADR-0011 framing unresolved; Hamel's "error-analysis-first" suggested classification.
- **Axes grilled:** Observability (which signals each surface), Failure modes (different inputs: staged diff vs full conversation trace), Reversibility (framing locks user mental model — getting it wrong forces a re-doc).
- **Key questions asked:** Do verify-output and postmortems run on the same input? (No — diff vs trace.) Do they output the same thing? (No — block-commit vs new-rules-discovered.)
- **Resolution:** DECIDED
- **Decision:** **Complementary at different loop positions.** ADR-0011 explicitly classifies: `verify-output` = static pre-commit check on staged diff for KNOWN slop classes; `chain-postmortem` = post-incident error-analysis on real conversation traces for NEW failure categories that feed back into verify-output's ruleset.
- **Spec update:** Slice 3 acceptance criteria adds: "ADR-0011 explicitly classifies verify-output (static, pre-commit, staged diff, KNOWN slop) and postmortems (error-analysis, post-incident, full trace, NEW categories) as complementary, not peer."

### Item Q4 — R5 implementation shape

- **Starting state:** Passive doc vs active skill — speculative whether users hit overflow.
- **Axes grilled:** Performance (1hr to write vs 1day to skill), Reversibility (passive → active is non-breaking; active → passive is breaking), Observability (how do we know overflow is happening?).
- **Key questions asked:** What's the empirical evidence rate of context-overflow in current chain runs? Answer: unknown, no postmortems yet. Therefore the active-skill version is premature.
- **Resolution:** DECIDED
- **Decision:** **Passive doc section in `using-habeebs-skill` for v1.10.0.** Revisit trigger: 3+ postmortems show context-distraction as the failure mode → promote to active skill in v1.11.0.
- **Spec update:** Slice 5 acceptance criteria pins to passive-doc; revisit trigger added to revisit-triggers section.

### Item Q5 — ADR-0004 amendment for Yan's principle

- **Starting state:** Implicit invariant (parent-context-sharing) not stated.
- **Axes grilled:** Reversibility (implicit invariants degrade over time as authors change), Observability (no test asserts the invariant).
- **Key questions asked:** Has any current dispatch violated the principle? Answer: no — Phase 1 context + decomposition + steering all flow to subagents per current dispatch templates. The risk is future drift.
- **Resolution:** DECIDED
- **Decision:** **One-line clause in ADR-0004 Decision section + Cognition AI citation.** ~3 sentences extending the existing dispatch contract: "Subagents MUST receive the parent's full context (Phase 1 context, decomposition, steering) — handoffs at this layer carry full state, not pointers (per Yan, Cognition AI, 'Don't Build Multi-Agents', 2025-06-12)."
- **Spec update:** Slice 2 expanded to cover both R2 (HANDOFF doc) AND ADR-0004 amendment. Slice 2 retitled.

### Item Q6 — R4 corpus design

- **Starting state:** Approach undefined; HITL gate on Slice 4.
- **Axes grilled:** Failure modes (Hamel's 100%-pass anti-pattern by construction if synthetic), Observability (manual grading rubric needed), Scale (30 prompts = ~45 min to curate; real-user log capture is OSS-impossible).
- **Key questions asked:** What deliberate adversarial cases prevent 100% pass? Answer: vague prompts ("add something"), wrong-skill bait ("research-sounding bug fix"), multi-skill-applicable scenarios, edge cases (typos, mixed signals).
- **Resolution:** DECIDED
- **Decision:** **Hand-curated 30-prompt mix: ~15 happy-path + ~15 adversarial/boundary cases.** Manual grading rubric: for each prompt, the expected triggered skill (or `[no-skill]`) recorded in the corpus; manual review documents which skill *actually* triggered when the description was read.
- **Spec update:** Slice 4 drops HITL label (the gate is closed). Slice 4 acceptance criteria specifies the 15/15 mix and adversarial-case categories explicitly.

### Item Q7 — Prompt-injection findings location

- **Starting state:** New file vs ADR-0004 extension vs new ADR.
- **Axes grilled:** Migration (where would future authors look for this?), Reversibility (split docs are hard to consolidate), Observability (one tiny file rots; section in existing ADR stays maintained).
- **Key questions asked:** Does ADR-0004 already cover untrusted-content rules? Answer: yes. Is this new evidence or a new rule? Answer: evidence for an existing rule.
- **Resolution:** DECIDED
- **Decision:** **Extend ADR-0004's existing untrusted-content section with a dated evidence paragraph.** ~2-3 sentences: "On 2026-05-13, three fabricated `<system-reminder>` tags surfaced in fetched HTML from `developers.googleblog.com`, `adk.dev`, and `code.claude.com` during prior-art-research dispatches. All ignored per this rule; documented as confirmation that the untrusted-content invariant is load-bearing in practice, not theoretical."
- **Spec update:** Slice 2 acceptance criteria includes ADR-0004 amendment with both the Yan clause AND the dated evidence paragraph.

### Item Q8 (factor-check F6) — R5 summary-and-flush schema

- **Starting state:** R5 spec said "active artifacts (current spec, current ADRs, current slice)" — not a concrete template.
- **Axes grilled:** Failure modes (sub-session reads thin summary and misses load-bearing state → re-asks human or fails silently), Reversibility (schema drift over time if not pinned), Performance (summary-writing cost vs sub-session ramp-up cost trade-off).
- **Key questions asked:** What's the minimum a fresh sub-session needs to continue work mid-chain? Answer: enough to skip a Phase 1 cold start.
- **Resolution:** DECIDED
- **Decision:** **Rich 7-section template:** (1) Active artifacts with file paths, (2) Current slice + acceptance-criteria status, (3) Last successful action, (4) What's blocking, (5) Open grill Qs from this session, (6) Recent test state, (7) Branch / worktree pointer.
- **Spec update:** Slice 5 acceptance criteria lists the 7 sections explicitly; ADR-0012 § "Summary template" provides the markdown skeleton.
