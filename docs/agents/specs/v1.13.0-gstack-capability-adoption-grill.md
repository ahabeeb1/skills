## Per-item resolutions

### Item D1 — `security-audit` trigger surface + trim depth

- **Starting state:** Spec offered standalone slash / conditional chain phase / both; full 14-phase vs trimmed.
- **Axes grilled:** Reversibility, Observability (discoverability of the trigger surface).
- **Key questions asked:** Is security review every-PR work? Does a conditional phase need a "security-sensitive" predicate that must itself be dogfood-tested?
- **Resolution:** DECIDED.
- **Decision:** **Standalone slash skill only** (`/security-audit`), invoked on demand — matches gstack `/cso`, no new chain phase or predicate to design. **Trimmed core** methodology: keep attack-surface census, secrets archaeology (git history), OWASP Top 10, STRIDE per-component, confidence-gated false-positive filter, markdown report. Drop cross-run trend-tracking/fingerprinting (needs persisted baseline state), heavy CI/CD/infra scans, JSON artifacts.
- **Spec update:** Slice 1 — remove the "optional conditional phase after verify-output" alternative; pin the trimmed-core phase list in acceptance criteria.

### Item D2 — `release` placement + hook awareness

- **Starting state:** Spec flagged the commit-block hook also blocking the release tag-push; offered chore-branch workaround / hook fix / disable-env-var.
- **Axes grilled:** Failure modes (release hits the block every run), Reversibility (one-way-ness of a hook change), Concurrency (n/a).
- **Key questions asked:** What exactly does the hook match? (Investigated: `*"git commit"*|*"git push"*`, blocked only on default branch — so `git push origin <tag>` on `main` after merge is the real misfire, not `git tag` itself.) Is blocking a tag-only push on main ever correct? (No — ADR-0001's rule is "no branch commits to default"; an annotated tag is not that.)
- **Resolution:** DECIDED.
- **Decision:** **Fix the hook.** Amend ADR-0003: refine `preventing-commits-to-default.sh` so a tag-only push (`git push origin <tag>`, `git push --tags`) is distinguished from a branch-commit push and allowed on the default branch. The `release` skill is a terminal chain link after `tdd-loop`. Slice 2 therefore also amends ADR-0003 and updates the PreToolUse hook dogfood.
- **Spec update:** Slice 2 — pin "fix the hook" as the mechanism; add the ADR-0003 amendment + `tests/dogfood/06-...`/hook dogfood update to acceptance criteria.

### Item D3 — `devex-review` trigger predicate

- **Starting state:** Spec said "mirror agent-factors-check" without pinning the predicate.
- **Axes grilled:** Scale (false-trigger rate across spec types).
- **Key questions asked:** Would "any public interface" fire on internal service APIs where DX isn't the point? (Yes — too broad.) Does "CLI/plugin only" miss SDKs? (Yes.)
- **Resolution:** DECIDED.
- **Decision:** Fires when the spec's product is **consumed primarily by developers** — CLI, SDK, library API, plugin, or developer framework. Mirrors `agent-factors-check`'s domain-gate shape.
- **Spec update:** Slice 3 — pin the predicate in acceptance criteria.

### Item D4 — Learnings-ledger shape + seeding

- **Starting state:** Spec proposed a new curated ledger extending `postmortems/`.
- **Axes grilled:** Reversibility, Observability, the "deletion test" / "boring backup" cross-cutting probe.
- **Key questions asked (user-initiated pushback):** What is the maintenance overhead of another curated doc? How would it actually improve skill feedback? Answer surfaced: gstack's `/learn` exists to thread memory across 10–15 parallel sprints — a problem habeebs-skill's one-chain-per-feature model does not have; the genuine learnings are already captured by ADRs, `postmortems/`, and SYSTEM_CONTEXT's reconciliation log; a manual ledger decays without the `/learn prune` automation ADR-0002 forbids.
- **Resolution:** DECIDED — **slice cut.**
- **Decision:** Drop Slice 4. The "cross-session learning" capability is not a real gap. Add a one-paragraph note to `using-habeebs-skill` stating cross-session learnings live in ADRs / `postmortems/` / SYSTEM_CONTEXT reconciliation log by design, so a future reader does not re-propose a ledger.
- **Spec update:** Remove Slice 4 entirely; portfolio ceiling drops to **3 new skills, 0 new conventions**. Add the `using-habeebs-skill` note as a small acceptance criterion on Slice 3 (or v1.14.0 bundle).

### Item D5 — Doc-sync scope

- **Starting state:** Spec offered fold-into-`release` / skip / separate skill.
- **Axes grilled:** Scope discipline (portfolio ceiling).
- **Resolution:** DECIDED.
- **Decision:** **Fold into `release`.** `release` runs a doc-sync sub-phase: map shipped features lacking docs + apply the CHANGELOG "sell test" (does the entry explain value, not just the change). No separate skill.
- **Spec update:** Slice 2 — add the doc-sync sub-phase to the description and acceptance criteria.

### Item D6 — Release sequencing

- **Starting state:** Spec provisionally named one `v1.13.0` covering all slices.
- **Axes grilled:** Reversibility, Failure modes (review/rollback surface of mixing a hook change with new skills).
- **Resolution:** DECIDED.
- **Decision:** **Stagger.** v1.13.0 = `security-audit` alone (highest-value gap, fully independent, zero hook risk). v1.14.0 = `release` + `devex-review` (the hook/ADR-0003 amendment is isolated and reviewable on its own).
- **Spec update:** Tag each slice with its target version; note the spec doc now spans two releases.

### Item Q1 — ADR-0003 hook coverage for a `/careful`-style guard

- **Starting state:** Open — "check before deferring safety primitives."
- **Resolution:** DECIDED (by investigation, no user input needed).
- **Decision:** The PreToolUse hook matches **only** `git commit`/`git push` and blocks only on the default branch. It does **not** cover `rm -rf`, `DROP TABLE`, `push --force`, etc. A `/careful`-style destructive-command guard is genuinely uncovered — and stays **out of scope** for this adoption (research already deferred it; it is marginal for a methodology plugin).
- **Revisit trigger:** A destructive-command incident appears in a `postmortems/` entry.

### Item Q2 — Porting discipline (gstack MIT)

- **Resolution:** DECIDED (procedural, no user input needed).
- **Decision:** Methodology and patterns are portable (not copyrightable). Every ported skill's text is written fresh, never copied from gstack's MIT-licensed files. The ADR credits gstack (garrytan/gstack) as the source of the adopted patterns.

### Item Q3 — Demand-validation (`/office-hours`) scope

- **Axes grilled:** Scope discipline.
- **Resolution:** DECIDED — **out of scope.**
- **Decision:** No upstream "is this worth building" gate. `prior-art-research` already anti-triggers on vague intent ("Do NOT use when intent is too vague — ask a clarifying question"). Demand validation is a founder/product concern, not engineering methodology — adopting it would be scope creep.
- **Revisit trigger:** 3+ chain runs that, in hindsight, should have been killed before research because the feature was not worth building.

### Item Q4 — Ledger seeding (automatic vs manual)

- **Resolution:** OUT-OF-SCOPE — moot. Slice 4 was cut (see D4); there is no ledger to seed.
