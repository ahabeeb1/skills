## Per-item resolutions

### Item 1 — OQ-1: Does removing "Make sure to use this skill" move the needle?

- **Starting state:** 650-trials study A/B-ed `Use when` (passive) vs `ALWAYS invoke` (directive). Did NOT test `Make sure to use this skill` (current habeebs phrasing) — plausibly that already saturates 80% of the imperative gain.
- **Axes grilled:** Reversibility, Performance (firing-rate is the metric), Observability.
- **Key questions asked:**
  - If post-release eval shows flat firing-rate, what's the rollback?
  - What's the actual numerator/denominator in the transcript eval?
  - "What would have to be true" for the rewrite to not work?
- **Resolution:** DECIDED.
- **Decision:** Proceed with Slice 3 directive-imperative rewrite. Lock the post-release success metric: **(sessions with any of `"build"`, `"add"`, `"refactor"`, `"fix this"`, `"design"`, `"implement"` in the user prompt AND the matched entry-point skill fires within 2 turns) / (sessions with any of those keywords)**, measured pre-release-baseline vs +30 days post-release. **Threshold: >10 percentage-point lift = success.** If <10pp, the v1.20.0 fallback is *not* a revert to current phrasing — it's a third variant: `You MUST use this skill when…` (imperative-with-pronoun, Brainstorming-style). That gives us 3 data points across the imperative-gradient.
- **Revisit trigger:** Post-release transcript eval at +30 days. If <10pp lift, file v1.20.0 candidate "imperative-with-pronoun rewrite."
- **Spec update:** Slice 6 acceptance criteria gains the explicit numerator/denominator and the 10pp threshold.

### Item 2 — OQ-2: Is `parallel-dev` truly an entry point?

- **Starting state:** Spec listed `parallel-dev` as 1 of 5 entry-point skills. Users rarely say "do these in parallel"; it's primarily invoked by other habeebs skills.
- **Axes grilled:** Failure modes, Reversibility (trivial — single frontmatter line).
- **Key questions asked:**
  - If demoted, what user-language case still works? What case breaks?
  - Who actually invokes `parallel-dev` in the codebase?
- **Resolution:** DECIDED.
- **Decision:** **Demote `parallel-dev`** with `disable-model-invocation: true`. Slash-command `/parallel` is preserved by the spec constraint. Internal callers (prior-art-research Phase 2.5 + Phase 4 Deep, tdd-loop AFK orchestration, write-plan parallelization-map consumer) invoke via explicit `Skill(parallel-dev)`. Math: 3 internal callers, 1 user-direct surface. The auto-invocable entry-point set drops from 5 to 4: `prior-art-research`, `systematic-debugging`, `deep-modules`, `security-audit`. Plus 3 support context skills (`using-habeebs-skill`, `setup-habeebs-skill`, `using-worktrees`). **Total auto-invocable: 7 skills.**
- **Revisit trigger:** If a postmortem ever cites "user said 'do these N in parallel' and parallel-dev failed to fire," reverse.
- **Spec update:** Layer 2 demoted-list grows from 10 to 11. Slice 2 acceptance criterion updated. Slice 5 routing-table entry for "do these N things in parallel" stays (now points to `/parallel`).

### Item 3 — OQ-3: Do `agent-factors-check` and `devex-review` get demoted?

- **Starting state:** Both are conditional extensions of `socratic-grill`. Spec leaned demote; OQ asked whether they should stay auto-invocable on narrow agent-spec / CLI-spec signals.
- **Axes grilled:** Failure modes (what trigger phrasing would actually fire them solo?), Concurrency (do they collide with `socratic-grill`?).
- **Key questions asked:**
  - What user phrasing triggers `agent-factors-check` and NOT `socratic-grill`?
  - Same for `devex-review`?
  - "Boring backup" probe.
- **Resolution:** DECIDED.
- **Decision:** **Demote both.** No realistic user phrasing triggers either skill standalone — users say "grill this spec," which triggers `socratic-grill`; the parent skill then invokes the conditional extension. Demoting makes the architecture honest about how they fire. They remain `/factor-check` and `/devex-review` slash-invocable.
- **Revisit trigger:** None expected — these are sub-skill-invoked by design.
- **Spec update:** Slice 2 description should explicitly call out *why* these two are demoted: they are sub-skill-invoked by `socratic-grill`, not user-invoked.

### Item 4 — OQ-4: Does YAML multi-line description truncation bite habeebs?

- **Starting state:** Research Case 7 claimed block-scalar (`|` / `>`) descriptions get truncated at line 1. habeebs descriptions span multiple visual lines.
- **Axes grilled:** Failure modes (silent truncation = worst kind), Observability (would we even know?).
- **Key questions asked:**
  - What's the actual YAML form of habeebs descriptions?
  - What's the canary against a future author accidentally introducing block-scalar form?
- **Resolution:** DECIDED — not a current bug.
- **Decision:** habeebs descriptions are **plain-scalar single-line** YAML (verified by inspecting `skills/prior-art-research/SKILL.md`). No `|` or `>` indicators. Apparent multi-line in editors is terminal wrapping only. This session's system reminder shows all 18 descriptions in full — further confirmation. **Bug doesn't apply.** To prevent future regression: Slice 4's `check-description-budget.sh` gains an assertion rejecting block-scalar indicators on any `description:` field.
- **Revisit trigger:** If a future Claude Code release changes YAML parsing behavior, re-audit.
- **Spec update:** Slice 4 acceptance criteria gains: "Reject SKILL.md files where the `description:` field uses block-scalar indicators (`|` or `>`)."

### Item 5 — OQ-5: `## Skill routing` block — inline in CLAUDE.md or separate file?

- **Starting state:** Spec leaned inline. Alternative was `docs/agents/SKILL_ROUTING.md` referenced from CLAUDE.md.
- **Axes grilled:** Performance (always-loaded budget), Reversibility (file moves are trivial).
- **Key questions asked:**
  - Token cost inline vs reference?
  - Anti-pattern check against Anthropic best-practices?
- **Resolution:** DECIDED.
- **Decision:** **Inline in CLAUDE.md**, placed after the `## The chain` block and before `## Triggering principles`. Pay ~250-400 tokens per turn in exchange for the routing being available at decision time (always-loaded). The alternative (separate file) is read only ~1-in-3 sessions, which defeats the purpose of routing primer.
- **Revisit trigger:** If CLAUDE.md grows past 200 lines (Anthropic-recommended ceiling for system-prompt density), split routing into `docs/agents/SKILL_ROUTING.md` and replace with one-line reference.
- **Spec update:** Slice 5 placement decision locked. Add the 200-line revisit-trigger to the spec's Revisit triggers section.

### Item 6 — OQ-6: Dogfood 13 successor — replace, augment, or sunset?

- **Starting state:** Dogfood 13 (synthetic-corpus trigger-precision) reports 34/34 (100%). User observes real-session failures. Hamel-Husain "synthetic prompts approach 100% by construction" red flag confirmed empirically. Slice 6 proposed wholesale replacement.
- **Axes grilled:** Observability (the eval IS observability for this feature), Migration (transition path), Reversibility.
- **Key questions asked:**
  - What's the transition path — hard cut, or both run in parallel?
  - What concrete cadence + rubric does the new eval need?
  - What if multiple eval runs overlap?
- **Resolution:** DECIDED.
- **Decision:** **Augment, then sunset.** v1.19.0+: both dogfood 13 and the new real-session transcript eval co-exist. Dogfood 13 becomes a "regression baseline" — re-run after Slice 3's description rewrite to confirm no precision *regression*. Real-session transcript eval (`docs/agents/references/trigger-firing-eval.md`) becomes the *primary* firing-rate signal. After 2 quarters of dual-tracking (v1.20.0+), if real-eval consistently surfaces failures dogfood 13 misses, sunset dogfood 13. The transcript eval is defined as: source = `~/.claude/projects/<repo-slug>/sessions/*.jsonl`; sample = last 30 sessions or last 14 days (whichever yields more); anonymization = redact user prompts to keyword-level summary; scoring = the OQ-1 numerator/denominator + per-skill recall; idempotency = pin sample-set with date-range header so re-runs are reproducible.
- **Revisit trigger:** v1.20.0 (2 quarters after v1.19.0 release).
- **Spec update:** Slice 6 acceptance criteria expands with pinned sample-set, anonymization rubric, idempotency. Dogfood 13 README updated to "regression baseline" (not "superseded").

## New decisions surfaced during grilling

The grill itself surfaced two decisions the spec didn't anticipate:

1. **Post-release success metric is now load-bearing.** The spec said "real-session transcript eval" but never defined the success threshold. Now locked at **>10 percentage-point improvement** in the numerator/denominator defined in OQ-1 resolution. Without this threshold, Slice 6 would emit vibes-based judgments.

2. **v1.20.0 fallback path is `You MUST use` (imperative-with-pronoun), not a revert.** The spec implied "if rewrite fails, rollback to current." Better path: a *third* variant on the imperative gradient that gives us 3 data points instead of 2. This makes the v1.19.0 → v1.20.0 transition exploratory-evolution rather than binary-revert.
