## Per-item resolutions

### Item 1 — OQ-1: Changeset aggregation placement in release flow

- **Starting state:** Spec recommended in-PR but flagged as load-bearing. Two options: (a) aggregate inside the release PR (bump commit lives in the PR), (b) aggregate post-merge (tag-time only, separate commit on main).
- **Axes grilled:** Failure modes, Reversibility, Concurrency.
- **Key questions asked:**
  - What does the v1.19.0 release skill currently do? — Answer: opens a release PR with `plugin.json` / `marketplace.json` bumps + CHANGELOG entry baked in; tag-push happens *after* merge via `git push origin refs/tags/<version>`.
  - If aggregation runs post-merge, who commits the bump to main? — Answer: would require a direct commit to main, which the `preventing-commits-to-default.sh` hook blocks. Would have to disable the hook for the release commit. Brittle.
  - If aggregation runs in-PR, what reverts cleanly if the PR is closed without merging? — Answer: changesets are simply not deleted; next release picks them up. The PR's bump commit goes away with the closed PR.
- **Resolution:** DECIDED
- **Decision:** **In-PR aggregation.** Release skill runs aggregation as the FIRST step of opening the release PR (read `.changeset/*.md` → compute new version → write `plugin.json` + `marketplace.json` + CHANGELOG.md → delete consumed changesets → commit all four mutations as one atomic commit → push branch → open PR). The tag-push step (after PR merge) remains unchanged from v1.19.0.
- **Spec update:** Slice 3 description clarified to specify "first step of release PR creation," with the explicit sequence above.

### Item 2 — OQ-2: Changeset deletion atomicity

- **Starting state:** Spec recommended single commit + git reset rollback. Question was the failure-mode shape.
- **Axes grilled:** Failure modes, Reversibility.
- **Key questions asked:**
  - What can fail mid-aggregation? — Filesystem write fails (disk full / permission), pre-commit hook rejects, git commit itself fails on signing.
  - What's the rollback signal? — Non-zero exit from the aggregation script.
  - What state survives a failed aggregation? — If script fails BEFORE the commit, working tree has partial edits + deleted changesets. If after the commit, the commit is itself the atomic unit.
- **Resolution:** DECIDED
- **Decision:** Single-commit, all-or-nothing semantics enforced at the script level. Aggregation script (`skills/release/scripts/aggregate-changesets.sh`) does ALL file mutations in a temp staging dir, then moves atomically. If any step fails: `git checkout -- .changeset/ plugin.json marketplace.json CHANGELOG.md` rolls back to pre-aggregation state. Exit codes: 0 = success + commit ready, 1 = aborted clean (no state change), 2 = aborted dirty (manual intervention required — should never happen given temp-dir approach but documented for safety).
- **Spec update:** Slice 3 acceptance criteria gains "atomic-or-rollback script behavior verified by dogfood scenario 23 with a forced-failure fixture."

### Item 3 — OQ-3: Two simultaneous PRs both bumping minor

- **Starting state:** Spec recommended bullet-list CHANGELOG aggregation. Question was the version-slot resolution.
- **Axes grilled:** Concurrency, Scale.
- **Key questions asked:**
  - Can two release PRs be open simultaneously? — In principle yes, in practice no: habeebs-skill is single-author with manual release. The race window is "PR A opens with bump to v1.21.0, PR B opens with bump to v1.21.0 before A merges." Each PR's aggregation only sees the changesets that exist when it runs.
  - What if both merge? — Whichever merges second hits a CHANGELOG.md conflict (existing `## v1.21.0` line already there) AND a `plugin.json` version conflict (already at 1.21.0). The merge fails loudly.
  - Is this worth pre-empting with a CI lock? — Substrate constraint: no CI. And the failure mode is loud (merge conflict), not silent (data loss). Cost of pre-emption > cost of conflict.
- **Resolution:** DECIDED
- **Decision:** **Accept the collision; rely on git's own merge-conflict failure.** Single-author scale means this race is hypothetical, not actual. If the user ever opens two release PRs simultaneously, the second merge fails loudly with a `CHANGELOG.md` + `plugin.json` conflict — the operator resolves by closing the second PR, re-running aggregation against the post-first-merge state, and the changesets that targeted v1.21.0 in the second PR now correctly target v1.22.0.
- **Spec update:** Slice 3 notes appended: "Two-PR race is hypothetical at single-author scale; substrate-free design accepts merge-time conflict resolution per Pattern D (audit memo) — conflicts pushed to merge-time, handed to human."

### Item 4 — OQ-4: Late-binding rename tiebreak

- **Starting state:** Spec recommended alphabetic slug order with logging.
- **Axes grilled:** Concurrency, Reversibility.
- **Key questions asked:**
  - When does the tiebreak actually fire? — Only when one release contains 2+ new ADRs (multiple changesets each writing `adr-<slug>.md`). Rare but possible.
  - Is alphabetic determinism enough? — Yes if every ADR slug is unique, which the convention guarantees (slugs are noun-phrases describing the topic).
  - What if two slugs ARE identical? — Spec said "edge case — unlikely." Push back: it's not unlikely if two parallel sessions write ADRs on related sub-topics with similar names. Need an explicit failure mode.
- **Resolution:** DECIDED
- **Decision:** Tiebreak: **alphabetic slug order** (deterministic). **Slug collision (two `adr-<identical-slug>.md` files)** halts the rename script with exit 2 and a message: "Two ADRs with identical slug `<slug>` cannot be assigned distinct IDs. Rename one before retrying." Per Pattern B, the operator picks a more-specific slug and re-runs. No silent merge.
- **Spec update:** Slice 1 acceptance criteria gains "slug collision halts loud, never silently merges or overwrites" + dogfood scenario 21 fixture for the collision case.

### Item 5 — OQ-5: Which PRs require a changeset?

- **Starting state:** Spec recommended "required if PR modifies skills/, hooks/, or docs/agents/adrs/; optional if only docs/agents/research/ or docs/agents/postmortems/."
- **Axes grilled:** Failure modes, Migration.
- **Key questions asked:**
  - Why those paths? — `skills/` and `hooks/` are the user-installable surface. `docs/agents/adrs/` are load-bearing for methodology. `research/` and `postmortems/` are durable but don't change behavior.
  - Edge cases? — What about `CLAUDE.md`, `AGENTS.md`, `README.md`, `CHANGELOG.md` itself? `tests/`? `docs/agents/SYSTEM_CONTEXT.md`? `docs/agents/specs/` and `docs/agents/plans/`?
  - Enforcement mechanism? — Either pre-push hook (loud, blocks contributor), or release-skill audit at PR-creation time (soft, blocks release).
- **Resolution:** DECIDED
- **Decision:** **Release-skill audit at PR-creation time (soft enforcement), with explicit path rules.** 
  - **Changeset REQUIRED if PR modifies any of:** `skills/`, `hooks/`, `.claude-plugin/`, `plugin.json`, `marketplace.json` (the last two should NEVER be edited directly post-v1.20.0, but the audit catches drift).
  - **Changeset OPTIONAL if PR modifies any of:** `docs/`, `CLAUDE.md`, `AGENTS.md`, `README.md`, `CHANGELOG.md` (the audit emits an INFO note but does not block).
  - **Changeset NEVER required if PR modifies only:** `tests/`, `.gitignore`, GitHub-meta files (`.github/`, `.gitattributes`).
  - Release skill at PR-creation reads `git diff --name-only main...HEAD`, applies the rules, halts with a clear message if a required path was modified without an accompanying `.changeset/<slug>.md`.
- **Spec update:** Slice 3 acceptance criteria gains "release skill audits paths-touched against required/optional/never matrix; halts on required-without-changeset." New dogfood scenario 25 (`tests/dogfood/25-changeset-required-check/`) tests the three classes.

### Item 6 — PT-1: No rename of existing ADRs (asymmetric scheme safety)

- **Starting state:** Spec said "existing ADRs NOT renamed (Pattern B / immutable path). Asymmetric ADR scheme (old: `NNNN-slug.md`, new: same final shape but went through `adr-<slug>.md` intermediate state). Mitigated by single-shape final form."
- **Axes grilled:** Reversibility, Migration, Scale.
- **Key questions asked:**
  - Is the final form actually identical? — Yes: at rest, every ADR file is `NNNN-<slug>.md`. The asymmetry is only in the *creation path* (some were born numbered, some were born unnumbered and renamed). Readers cannot tell the difference.
  - Does any tool need to know the creation history? — Searched: no. `decision-record` reads existing ADRs but doesn't care how they were created. `release` skill's renamer cares only about `adr-*.md` files (new ones) vs `[0-9]*-*.md` files (already numbered). The two sets are disjoint and the convention enforces it.
  - What if the user accidentally writes `0020-foo.md` directly (skipping the late-binding step)? — Should be caught by `decision-record` skill template. Add a dogfood test.
- **Resolution:** DECIDED
- **Decision:** Asymmetric creation path is safe long-term. **No existing ADR (0001-0019) is ever renamed.** New rule for `decision-record` skill: writes go to `adr-<slug>.md` only; never write `NNNN-<slug>.md` directly (the release skill is the sole writer of numbered names). Dogfood scenario 21 gains an assertion: `decision-record` outputs never match `^[0-9]{4}-` pattern.
- **Spec update:** Concrete-picks table row "Existing ADRs (0001-0019)" — append "Note: `decision-record` skill MUST be the only writer of `adr-*.md`; `release` skill MUST be the only writer of `NNNN-*.md`. Separation enforced by dogfood scenario 21."

### Item 7 — PT-2: Slug-based vs random-id changeset naming if a second author joins

- **Starting state:** Spec chose slug-based on single-author grounds, said "branch-slug is already unique per branch."
- **Axes grilled:** Scale, Concurrency.
- **Key questions asked:**
  - When does slug-based break? — Two PRs whose branch slugs both shortened to the same changeset filename. E.g., `feature/fix-the-bug` and `fix/the-bug` both generate `the-bug.md`.
  - How likely with two authors? — Low but non-zero. Two devs working in parallel on overlapping topics could pick similar slugs.
  - What's the cost of moving to random-id later? — Trivial: just change the scaffold-generation step in `decision-record`-equivalent for changesets. Existing slug-based changesets keep working; the directory accepts both.
- **Resolution:** DECIDED
- **Decision:** **Ship with slug-based naming.** Add a revisit trigger: "if a second author joins the project AND the project sees ≥3 changeset filename collisions in any 90-day window, switch the changeset-scaffold step to random-id." Recorded in spec's Revisit triggers.
- **Spec update:** Revisit triggers — append the second-author/collision-count condition.

Pattern B's "immutable path" applies to the ADR files themselves, not to the directories the ADRs declared. ### Item 11 — DX-1: First-time contributor creates a changeset — what command / scaffold?

- **Starting state:** Spec mentioned `.changeset/<slug>.md` files but didn't specify how they're created.
- **Axes grilled:** Onboarding friction (DX).
- **Key questions asked:**
  - Does Changesets-the-tool have a CLI? — Yes, `pnpm changeset` opens a prompt. We can't use that without taking on the Node dependency, which violates ADR-0002.
  - What's the minimal substrate-free affordance? — A markdown template at `.changeset/README.md` with a copy-paste skeleton + an example file (`example.md`). Contributors `cp .changeset/example.md .changeset/<my-slug>.md` and edit.
  - Could `decision-record` skill be the scaffolder? — It writes ADRs, not changesets. Different concern, different artifact.
- **Resolution:** DECIDED
- **Decision:** **No CLI scaffolder; markdown skeleton at `.changeset/README.md` + `.changeset/EXAMPLE.md`** (uppercase to avoid being mistaken for an actual changeset). Contributors copy EXAMPLE.md and edit. The release skill audits frontmatter at PR-creation time so malformed changesets fail loud (per DX-4 below). YAGNI on a CLI tool — habeebs-skill is markdown-only and the cp-and-edit flow is exactly two commands.
- **Spec update:** Slice 2 acceptance criteria gains `.changeset/EXAMPLE.md` requirement with a fully-populated minimal example showing the `bump`/`why` frontmatter format + a one-line why explanation. **NEW DECISION SURFACED — minor scope add to Slice 2.**

### Item 12 — DX-3: Helper script ergonomics

- **Starting state:** Spec mentioned `skills/release/scripts/assign-adr-ids.sh` and `skills/release/scripts/aggregate-changesets.sh` but didn't specify invocation pattern.
- **Axes grilled:** API/CLI ergonomics (DX).
- **Key questions asked:**
  - Are scripts invoked by `release` skill in the prompt context, or by the user directly? — The release skill walks the user through a sequence; the user runs `bash skills/release/scripts/...` themselves (consistent with how v1.19.0 dogfood scripts work).
  - Can scripts be invoked standalone for debugging? — Yes, both should be safe to invoke against the current repo state.
  - Do scripts need flags? — Aggregation needs `--dry-run` (print what WOULD happen without committing). Late-binding rename needs `--dry-run` (print rename plan). Both should have `--help`.
- **Resolution:** DECIDED
- **Decision:** Scripts are user-invoked via `bash skills/release/scripts/<name>.sh`. Both scripts support `--dry-run` and `--help`. The `release` skill SKILL.md walks the user through the right invocation order. Scripts are safe to invoke standalone (no global state mutation; everything happens via git commit which is reversible).
- **Spec update:** Slice 1 + Slice 3 acceptance criteria gain "script supports --dry-run + --help flags."

### Item 13 — DX-4: Error messages

- **Starting state:** Spec was silent on error-message quality.
- **Axes grilled:** Error-message quality (DX), Failure modes.
- **Key questions asked:** What does each failure mode print?
  - Malformed changeset (missing `bump`): "Changeset `.changeset/<file>.md` missing required `bump` frontmatter field. Expected one of: patch, minor, major. See .changeset/EXAMPLE.md."

- **Resolution:** DECIDED
- **Decision:** All five error messages above are required in the helper scripts + release skill audit. Each names the offending file, the missing/wrong field, the valid values, and a pointer to documentation. **Dogfood scenario 25 fixtures each of the 5 failure modes and asserts the exact error message format.**

### Item 14 — DX-6: Upgrade friction for in-flight branches

- **Starting state:** Spec was silent on what happens to branches mid-flight when v1.20.0 ships.
- **Axes grilled:** Migration (DX), Failure modes.
- **Key questions asked:**
  - What does an in-flight branch look like pre-v1.20.0? — Possibly has manual edits to `plugin.json` + `CHANGELOG.md` already (the old workflow).
  - Does the release skill on main reject? — The release-skill audit (DX-4) requires a changeset for `skills/` modifications. If the in-flight branch modifies skills, it'd be rejected.
  - Mitigation? — Two options: (a) hard cutover (post-v1.20.0, every branch MUST have a changeset; in-flight branches must add one before merging), (b) soft cutover (audit emits a WARN, not a block, for the first 30 days, then escalates to block).
  - Substrate constraint: who would enforce a 30-day grace window? — Date-based logic in the release skill. Cheap to add but adds complexity for a one-time migration.
- **Resolution:** DECIDED
- **Decision:** **Hard cutover.** In-flight branches add a `.changeset/<slug>.md` before merging. The audit emits a clear error message pointing at `.changeset/README.md`. Single-author scale + manual release process makes a grace window unnecessary; the user IS the release process and the migration is one-time. Document the cutover in the v1.20.0 CHANGELOG entry: "Migration note: any open PR modifying `skills/`, `hooks/`, or `.claude-plugin/` must add a `.changeset/<slug>.md` before merging."
- **Spec update:** Slice 7 (v1.20.0 release) acceptance criteria gains "CHANGELOG entry includes migration note for in-flight branches."

## ADR candidates

High-impact decisions that should be captured in `decision-record`:

1. **`adr-late-binding-and-changesets`** — combines: (a) Backstage late-binding for new ADRs, (b) `.changeset/<slug>.md` append-only intent files, (c) `release` skill as the single coordinator (aggregation + ADR-int assignment both run in the release PR), (d) separation-of-writers (`decision-record` writes `adr-*.md`, `release` writes `NNNN-*.md`), (e) hard cutover migration policy.
