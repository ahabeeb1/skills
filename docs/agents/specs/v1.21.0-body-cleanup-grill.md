## OQ-A1 — Origins sections fate

### Starting state
verify-output/SKILL.md has `## Origins` attributing design to oh-my-claudecode's ai-slop-cleaner + ultraqa + repo's CLAUDE.md. Zero peer SKILL.md files in research had Origins sections.

### Audit recommended
(b) Move to `docs/agents/skill-origins.md` registry.

### Dimensions grilled
**Reversibility** + **Migration** + **Devex (docs-as-experienced)**

### Grilling

**Q: Is there ANY peer plugin precedent for a centralized origins registry?**

A: No. Research sampled 11 peer SKILL.md files (obra, mattpocock, oh-my-claudecode) + 7 Anthropic-canon files + 9 outside-CC instruction files. Zero of those 27 ship a `skill-origins.md` or equivalent attribution registry. Anthropic's skill-creator.md (the meta-skill teaching others how to write skills) doesn't even mention attribution as a concept.

**Q: If (b) is Modie-inventing-a-convention, what's the cost of carrying it?**

A: A new file at `docs/agents/skill-origins.md` joins the doc landscape that ADR-0009 already governs (3-consumer threshold for methodology files). With 1 attribution entry (oh-my-claudecode → verify-output) it'd be a 1-consumer file — fails the threshold. Pattern G from the v1.19.0 audit (methodology folders earn existence by file count) applies: a 1-entry registry doesn't earn its existence.

**Q: Does the courtesy-attribution argument carry enough weight to override convention?**

A: The CLAUDE.md attribution in the current Origins section is redundant (CLAUDE.md is the project's own canon — attributing the skill to its own repo is not real attribution). The `agent-factors-check applied to verify-output's own design` line is self-referential — also redundant. Only the oh-my-claudecode reference is a genuine external attribution. ONE attribution doesn't earn a section. ONE attribution can live in the commit message of v1.21.0 cleanup (`git log --all --grep="oh-my-claudecode"` will surface it forever) AND in a one-line `**Inspired by:**` mention inside a `<!-- author note -->` HTML comment in the SKILL.md if the author feels strongly.

**Q: What about future skills that might have multiple genuine attributions?**

A: That's the revisit trigger. If `≥3 skills` need genuine external attribution (not self-reference, not CLAUDE.md), revisit and create the registry then. Today: 1 skill, 1 real attribution. Not enough.

### Resolution
**DECIDED → (a) delete the Origins section in verify-output.**

- Attribution to oh-my-claudecode preserved in the v1.21.0 commit message (durable in `git log`)
- Optional: add an HTML-commented `<!-- Inspired by oh-my-claudecode/ai-slop-cleaner + ultraqa -->` line near the top of verify-output/SKILL.md body (Anthropic best-practices doesn't forbid HTML comments, dogfood scenarios 26-28 will exclude HTML-commented regions)
- Revisit trigger: **3+ habeebs-skill skills need genuine external attribution** (excluding self-reference and CLAUDE.md), promote to `docs/agents/skill-origins.md` registry

### Audit amendment
The audit recommendation (b) is overruled. Per the 27/27 convention + the 3-consumer threshold + Pattern G, (a) wins. The audit gets a 2026-05-26 amendment noting this.

## OQ-A2 — External research-paper citations supporting load-bearing quantitative claims

### Starting state
parallel-dev/SKILL.md lines 45-48 footer:
- Anthropic's "Built a multi-agent research system" — supports the ~90% lift / 15x token cost claim
- Cognition's "Don't build multi-agents" — supports the write-task failure-mode rule
- v1.19.0 internal audit memo — backward-citation to internal artifact

### Audit recommended
Keep the two external citations; delete the internal audit-memo citation.

### Dimensions grilled
**Reversibility** + **Devex (defensibility for future contributors)**

### Grilling

**Q: Does the 27/27 research convention distinguish external research papers from internal ADRs?**

A: The research extraction template tracked "inline ADR-NNNN citations" specifically (Pattern A). It did NOT directly score "external research-paper citations." Looking at the underlying data:
- Anthropic's skill-creator.md cites zero external papers in body
- obra/superpowers's test-driven-development.md cites no external papers
- Cline's `.clinerules/general.md` cites one PR (#7566)
- The closest analog to "external paper citation" in the 27-file sample is **zero**

So technically the research found "no external-paper citations either" — but the sample skews toward task-instruction skills with no empirical claims. parallel-dev IS unusual: it cites a 90% lift / 15x cost claim that an operator might be skeptical of.

**Q: What's the boring backup version of this decision?**

A: Drop the quantitative claim entirely. Restate the section to be qualitative: "Research-shaped tasks see substantial lift at meaningful token cost — the cost is real, the gain is real" (no number, no source). This is the cleanest application of the 27/27 rule, but it abandons the operational value of the specific numbers an operator can quote.

**Q: What's the asymmetric-risk version?**

A: Keep the external links because they back load-bearing quantitative claims AND they pass the spirit of the 27/27 rule (which targeted internal backward-citations as cruft, not external authoritative-source links). Delete the internal audit-memo bullet because it's classic backward-citation-to-internal-artifact. Codify the rule.

**Q: Future agent author reading parallel-dev — do they need the citations?**

A: Yes when they're defending the empirical claim to a skeptic ("how do you know 90% lift?"). No when they're operating the skill (instructions stand without sources). The citations serve a defensibility-of-the-rule function distinct from a how-to-use function.

**Q: Devex axis — when a skeptic asks "why 8 concurrent?" or "why this number," is the path to the source one click or three?**

A: With external citations as Pattern-D forward-only footer-pointers, one click. Without, the skeptic has to grep git history or read the v1.19.0 audit memo (which the rule also forbids citing). The Devex cost of full deletion is too high.

### Resolution
**DECIDED → keep external research-paper citations, drop internal audit-memo citation.**

The codified rule for the cleanup:

> **Pattern-D exception for empirical claims:** A `## Sources for this section:` footer (or similar) is permitted when the body contains a load-bearing quantitative claim (a specific number, percentage, or rate) that an operator might need to defend. Each footer entry MUST be a forward link to an external authoritative source (peer-reviewed paper, vendor engineering blog, canonical man page) — never to an internal ADR, postmortem, audit memo, or in-repo document. Forward links to in-repo *operational* artifacts (templates, reference docs) are governed by Pattern D directly and don't need this exception.

This rule applies to OQ-A3 too.

### Audit confirmation
Audit recommendation stands. The grill codified the rule that justifies it.

## OQ-A4 — See-also entries that link to ADRs with operational guidance in the description

### Starting state
Multiple See-also blocks have entries like `- [ADR-0011](path) — when verify-output keeps missing a failure class, postmortems are the canonical place to...`. Pattern D says forward-only; the ADR link is backward; but the description carries useful operational prose.

### Audit recommended
Convert each `[ADR-NNNN]` link target to the most-operationally-useful forward link (`[docs/agents/postmortems/]` for ADR-0011's case), preserve the description prose.

### Dimensions grilled
**Failure modes** (the lookup-the-reasoning failure mode) + **Devex (docs-as-experienced)** + **Reversibility**

### Grilling

**Q: Is "go look at the postmortems dir" actually semantically equivalent to "go read ADR-0011 about error-analysis cadence"?**

A: No. The postmortems directory is the *output store* (what postmortems exist); ADR-0011 is the *cadence rule* (when and how to write them). A reader who follows the converted link sees N postmortem files but might miss the rule for *when* to write a new one.

**Q: Does that matter operationally?**

A: Two reader-archetypes:
1. **Skill user**: agent or human reading verify-output's See-also during a verify-output run. They want "where do I look when I find a failure verify-output missed?" The postmortems dir + `README.md` is the right answer (`README.md` has the template, including the trigger rules). ADR-0011 is overkill.
2. **Skill author / methodology contributor**: someone reading verify-output's See-also to understand *why* postmortems exist. They want ADR-0011 directly. But this archetype isn't reading See-also — they're reading the ADR index.

Reader-archetype 1 dominates the See-also reader path. Convert to forward link.

**Q: What if `docs/agents/postmortems/README.md` doesn't have the trigger rules?**

A: Verified — it does. Postmortem README contains the template per ADR-0011 + the "what postmortems produce" rule + trigger conditions. The forward link to the postmortems dir lands on a README that contains the operational rule the converted See-also description references.

**Q: What's the reversibility cost?**

A: Trivial. If a future reader complains they couldn't find the ADR rationale, add an ADR index back-pointer in `docs/agents/postmortems/README.md` itself (one edit, one file, no skill-body change). Or add a v1.22.0 amendment that restores the ADR link if the operational forward-link truly fails.

**Q: Devex axis — does this conversion lose anything for the user types we care about?**

A: No. The user-experiencing-the-See-also is the operator, not the methodology researcher. Operator wants pointers to what to DO next, not citations of what to READ for justification.

### Resolution
**DECIDED → convert all `[ADR-NNNN]` See-also link targets to the most-operationally-useful forward link, preserving the description prose.**

Concrete conversions (audit will be updated with the table):

| Original See-also entry | Converted forward target |
|---|---|
| `[ADR-0011]` (error-analysis cadence) | `docs/agents/postmortems/` |
| `[ADR-0004]` (parallel-subagent-dispatch contract) | `skills/parallel-dev/SKILL.md` § Return contract |
| `[ADR-0008]` (verify-output skill scope) | Delete entry entirely (scope already in skill body) |
| `[ADR-0007]` (description policy) | `tests/dogfood/11-description-budget/` |
| `[ADR-0015]` (hook tag-push allowance) | `hooks/` directory |
| `[ADR-0003]` (hooks scope) | `hooks/` directory |

The ADR index at `docs/agents/adrs/README.md` continues to list every ADR with its `## Affects:` back-reference — that's the canonical reverse-lookup. The skill-body forward links are operational.

### Audit confirmation
Audit recommendation stands; the grill validated the rule.

## OQ-A5 — Version tag in section headings

### Starting state
release/SKILL.md has `### Phase 3.25 — Changeset aggregation + path audit (v1.20.0+)` and `## Description-policy audit (v1.19.0+)`.

### Audit recommended
Strip the `(v1.20.0+)` and `(v1.19.0+)` parentheticals from headings.

### Dimensions grilled
**Migration** (what does a contributor on an old version see) + **Devex (upgrade friction)**

### Grilling

**Q: Does removing the version tag from the heading hide useful information that an operator might need ("when did this phase appear")?**

A: Two readers:
1. **Operator running release in 2026-05+**: they have the current SKILL.md from the plugin install. The phase EXISTS. They don't need to know it appeared in v1.20.0. The version tag is noise.
2. **Methodology archaeologist asking "when did Phase 3.25 appear?"**: they look at `git log -p skills/release/SKILL.md` or the CHANGELOG. `git log --diff-filter=A --follow -p skills/release/SKILL.md | grep "Phase 3.25"` returns the introducing commit in 1 second. CHANGELOG.md has the v1.20.0 entry explaining the late-binding-and-changesets ADR.

Both readers are served without the version tag.

**Q: 5-whys probe — why is the version tag in the heading at all?**

A: Habit from when ADRs were getting amended in-place and we wanted body-readers to know "this is the new way." But ADR-0021 (folder cuts) already established the convention that amendments live in dated amendment sections within the ADR file, NOT inline in skill bodies. The version tag in the heading is residue from a prior convention.

**Q: Devex — upgrade friction. A contributor upgrading from v1.18 to v1.20 — do they need the tag to know what's new?**

A: No. CHANGELOG.md is the upgrade artifact. If they're reading release/SKILL.md *during* an upgrade, they want the current procedure, not a list of what's new since their previous version.

### Resolution
**DECIDED → strip version tags from all section headings.**

Apply to:
- `### Phase 3.25 — Changeset aggregation + path audit (v1.20.0+)` → `### Phase 3.25 — Changeset aggregation + path audit`
- `## Description-policy audit (v1.19.0+)` → `## Description-policy audit`
- `## Auto-invocation scope (v1.19.0+ per ADR-0007 § C)` (using-habeebs-skill) → `## Auto-invocation scope`
- Any other heading carrying a parenthetical version tag (audit script will surface; manual scan confirms these are the 3 major instances)

### Audit confirmation
Audit recommendation stands.

## OQ-A6 — Worst-offender semantic-load probes (3 specific restatements)

### Starting state
3 specific restatements from the 13-hit files. Pressure-test each for load-bearing semantics.

### Probe 1 — using-habeebs-skill Hit 1 (line 39 paragraph)

**Original:** Big paragraph attributing the full-doc-read contract to OpenAI's Agents SDK + Cognition's "Don't Build Multi-Agents" + ADR-0004.

**Restatement:** Keeps the rule; drops 3 citations.

**Pressure-test (Devex / defensibility):** Does dropping OpenAI + Cognition attribution weaken the rule's authority for a future agent author?

**Grilling Q: Is the OpenAI Agents SDK link operationally needed in the skill body?**

A: The Agents SDK "Handoff = ownership transfer" primitive is the SAME shape we use. The link is "this is a known-good pattern, not a Modie-invention." That's a defensibility-of-the-rule function (like OQ-A2's empirical claims) but **without a quantitative claim** to anchor on. The full-doc-read rule is qualitative: "read the full doc, not just the HANDOFF line."

**Q: If the rule itself ("read full doc, not just HANDOFF") is qualitative and self-evident from the failure mode it prevents, do we need external attribution?**

A: No — the failure-mode argument carries the rule. The rule's "wait, why is this true?" defense is the next paragraph in the file: "subagents on degraded context silently encode conflicting interpretations of the parent task." That's mechanism. Mechanism beats authority for a qualitative rule.

**Q: But what if a contributor pushes back?**

A: Convergence between OpenAI's published primitive + Cognition's published warning is genuine external corroboration. **However**: per the OQ-A2 codified rule, that corroboration only earns a footer slot if it backs a load-bearing quantitative claim. The full-doc-read rule has no number — it's a discipline. So the OpenAI/Cognition citations DON'T qualify under the OQ-A2 exception. They get dropped.

**Resolution for Probe 1:** Restatement stands. The rule lives on its own mechanism.

### Probe 2 — using-habeebs-skill Hits 8+9 (Standalone by design heading + closing link)

**Original:** `## Standalone by design (ADR-0002)` + `See docs/agents/adrs/0002-habeebs-skill-standalone.md.` trailing link.

**Restatement:** `## Standalone by design` + no trailing link.

**Pressure-test:** Does removing the ADR-0002 anchor make the section's load-bearing-constraint status less discoverable for a new contributor?

**Grilling Q: How does a new contributor discover that "no runtime services" is a hard constraint (not a preference)?**

A: Three paths:
1. The section title "Standalone by design" + the section body which says "one-time-use per feature, with no runtime dependencies" — that's the rule, stated directly
2. The CLAUDE.md at repo root lists ADR-0002 explicitly in the methodology overview
3. ADR-0002 itself in the ADR index has Status: Accepted

Removing the inline link from this skill body removes path 1's *citation footer* but not the rule itself. Paths 2 and 3 still work.

**Q: Discovery cost — for a new contributor who reads the SKILL.md first without going to CLAUDE.md?**

A: They see "Standalone by design" + "no runtime dependencies" — that's clear. If they want the deeper why ("why was this decided?"), they have to navigate to ADR-0002. Without the inline link, that's 1 extra step (open the ADR index, find ADR-0002).

**Q: Is that cost too high?**

A: No. The same contributor will eventually read the ADR index when they touch any methodology change — the index IS the right place to start for "why this was decided." The skill body's job is "what is the rule now," not "what reasoning produced the rule."

**Resolution for Probe 2:** Restatement stands. Heading + section body carry the rule; ADR-0002 lives in the index for the "why."

**Original:** "prior-art-research is the canonical SYSTEM_CONTEXT.md writer per ADR-0005 single-writer invariant"

**Restatement:** "prior-art-research is the canonical SYSTEM_CONTEXT.md writer"

**Pressure-test:** Is "per ADR-0005" semantically load-bearing, or just a provenance tag?

**Q: 5-whys — why is "per ADR-0005" there at all?**

A: To anchor "this rule is locked, not opinion." But the next 3 sentences in the same paragraph already say "single source of truth" and "every chain skill reads but only prior-art-research writes." The anchor is redundant with the prose that follows.

**Q: Future contributor question: "what happens if I just write to SYSTEM_CONTEXT.md from setup-habeebs-skill?"**

A: setup-habeebs-skill explicitly invokes Phase 0 rather than writing directly — that's stated in setup-habeebs-skill/SKILL.md Phase 7. The constraint is enforced *by the skill that would violate it*, not by a citation in prior-art-research/SKILL.md.

**Resolution for Probe 3:** Restatement stands. Rule preserved as direct statement; ADR cite was a provenance tag.

**Original:** "The chain's bleeding pain (documented 2026-05-12) was a research run that missed `hooks / event handlers` and `subagent-driven patterns`"

**Restatement:** "Without a coverage gate, the chain proceeds against incomplete decompositions and ships specs missing whole architectural axes (hooks / event handlers / subagent-driven patterns being the most common omissions)"

The "for example" framing keeps the categories as illustrations rather than empirical-frequency claims.

**Resolution for Probe 2:** Restatement needs refinement. Audit gets a 2026-05-26 amendment: prior-art-research Hit 3 final text is the "for example" version above.

### Probe 3 — deep-modules Hit 1 ("ADR-NNNN" → "<ADR title>")

**Original:** "contradicts ADR-NNNN — only worth reopening because…"

**Restatement:** "contradicts <ADR title> — only worth reopening because…"

**Pressure-test:** Is "<ADR title>" actually a clearer placeholder for an author writing inline, or does it read worse than "ADR-NNNN"?

**Grilling Q: What does the author actually want to write in this slot during real use?**

A: "contradicts the standalone-by-design ADR" or "contradicts the single-writer invariant ADR" — they want the rule's name, not the integer.

**Q: Is "<ADR title>" the right placeholder for that?**

A: It's better than "ADR-NNNN" because it suggests the right shape of substitution (a meaningful name, not a number). But "<ADR title>" is itself meta-syntax — a contributor might wonder if they're supposed to write the literal ADR file's `# Title` heading or a descriptive phrase.

**Q: Is there a cleaner placeholder?**

A: `<ADR title or short name>` is clearer. Or just rewrite the anti-pattern example to be concrete: "contradicts a locked ADR — only worth reopening because…" — no placeholder at all; the contributor writes "the standalone-by-design ADR" or "ADR-0002" or whatever they prefer; the example shows the SHAPE of the contradiction note, not a template to fill in.

**Q: Which wins for Devex?**

A: The concrete rewrite. Placeholders invite over-thinking. The example's job is to show the anti-pattern, not to be a template.

**Resolution for Probe 3:** Restatement needs refinement. Audit gets a 2026-05-26 amendment: deep-modules Hit 1 final text becomes:

> "Re-litigating decisions captured in ADRs — if a deepening contradicts a locked ADR, surface it but mark clearly ('contradicts the X ADR — only worth reopening because…')"

### Overall OQ-A7 resolution
**DECIDED with refinements on 2 of 3 probes:**
- Probe 1 (parallel-dev): restatement stands as audited
- Probe 2 (prior-art-research dated incident): refined to use "for example" framing
- Probe 3 (deep-modules placeholder): refined to use concrete framing without placeholder

Audit will be amended with the refined texts under "## 2026-05-26 Amendment."
