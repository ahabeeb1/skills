## TL;DR

Strip three cruft categories from 12 affected SKILL.md bodies — inline ADR-NNNN citations (remove + restate), inline version tags (remove wholesale), and dated incident references (move to `docs/agents/postmortems/`). Adopt a "behavioral-only body, provenance lives elsewhere" rule modelled on Anthropic's skill-creator + best-practices canon (27/27 cross-population convergence). 7 vertical slices; ~3-4 sessions; load-bearing audit-first deliverable in Slice 1 (per-file change-list with before/after + semantic-fidelity column) — user reviews + approves BEFORE any cleanup edit lands.

## Architecture

The cleanup is markdown-only (no runtime substrate — ADR-0002). Two artifact planes:

- **Body cleanup plane:** 12 SKILL.md files under `skills/<name>/SKILL.md`. Three regex-detectable cruft patterns get inventoried by an audit script, reviewed by the user in a single audit report, then mechanically applied file-by-file in dependency-ordered slices.
- **Postmortem migration plane:** dated incident narrative (rare — likely 2-4 hits total) moves to `docs/agents/postmortems/<date>-<slug>.md`. Directory already exists per ADR-0011 with one prior entry (`2026-05-12-missed-architectural-categories.md`).
- **Regression-prevention plane:** three new dogfood scenarios (26/27/28) detect re-introduction of any cruft category in future PRs.

```
                    [audit-script] → docs/agents/specs/v1.21.0-body-cleanup-audit.md
                          ↓                              ↓
              regex-detect cruft hits          ←── USER APPROVAL GATE ──→
                          ↓                              ↓
[Slice 2] tracer cleanups (lowest-hit files)  [Slice 5] postmortem migration
[Slice 3] mid-tier cleanups                   [Slice 6] dogfood scenarios 26-28
[Slice 4] worst-offender cleanups (13-hit)
                          ↓
              [Slice 7] self-dogfood v1.21.0 release via v1.20.0 changeset mechanism
```

## Concrete picks (from research)

| Decision | Choice | Reason |
|---|---|---|
| Inline ADR-NNNN citations | Remove wholesale; restate rule in present-tense imperative form | Pattern A 27/27; ADR index already carries back-references |
| Inline version tags ("Added in v1.X") | Remove wholesale; CHANGELOG.md is the canonical version log | Pattern B 27/27; Anthropic prescription verbatim |
| Dated incident references | Move narrative to `docs/agents/postmortems/<date>-<slug>.md`; restate surviving rule in body without date | Pattern C 26/27; postmortems/ already exists per ADR-0011 |
| Footer policy | `## See also` forward-only; present only on skills with ≥3 sibling-references or a bundled REFERENCE.md; otherwise omit | Pattern D — footers are rare and strictly forward-pointing |
| Cross-skill inline links | Keep `/skill-name` and `@sibling.md` handles in prose; don't convert to numbered cites | Pattern F — lightweight handles are the universal idiom |
| `<details>` "Old patterns" disclosure-fence | DO NOT adopt as default; reserve for genuinely-rare migration cases | Anthropic prescribes it as last-resort, not default |
| Length ceiling | Enforce ≤500 lines per SKILL.md; if exceeded, split into `skills/<name>/references/*.md` and forward-point | Pattern E — Cursor + Anthropic hard cap |
| Audit-first deliverable | Per-file change-list with before/after snippets + proposed action + semantic-fidelity column | User explicit instruction: "audit first, then decide" |
| Audit report location | `docs/agents/specs/v1.21.0-body-cleanup-audit.md` | Implementation artifact, not research; lives next to this spec |
| PR batching | 1 audit-report PR + ~3 cleanup-batch PRs (one per slice 2/3/4) + 1 release PR | Keeps reviewable surface bounded; not 12 separate PRs |
| Origins sections fate (grill OQ-A1) | DELETE entirely (not migrate to registry) | 27/27 peer convention has no Origins precedent; 1-entry registry fails ADR-0009 threshold; attribution preserved in v1.21.0 commit message; optional HTML-commented attribution line allowed |
| Empirical-claim sources footer (grill OQ-A2/A3) | Pattern-D exception: `## Sources for this section:` footer permitted IFF body has load-bearing quantitative claim; external authoritative sources only — never internal ADRs/postmortems/audit memos | parallel-dev (~90% lift claim) + using-worktrees (git-worktree man page) qualify; the v1.19.0 audit-memo line drops |
| See-also ADR-link conversion (grill OQ-A4) | Convert each `[ADR-NNNN]` See-also link target to operationally-useful forward link; preserve description prose | Audit's conversion table is canonical; ADR index `## Affects:` back-reference is the reverse-lookup |
| Version tag in section headings (grill OQ-A5) | Strip all `(vX.Y.Z+)` parentheticals from headings | Phase EXISTS now; CHANGELOG carries upgrade narrative |

## Trade-offs accepted

- **In-file "why does this rule exist" provenance is lost from the body.** Recovery: `git blame skills/<name>/SKILL.md` + `docs/agents/adrs/README.md` index.
- **Discoverability of ADRs from the skill that implements them is reduced.** Mitigation: ADRs list affected skills in their `## Affected skills` section; the back-reference exists, just not in the SKILL.md.
- **Per-skill version-introduced metadata at the rule level disappears.** Mitigation: CHANGELOG.md carries per-version skill deltas; git tag ↔ commit-hash chain reconstructs introduction date for any rule.
- **Restated rules require manual fidelity verification.** Audit report's semantic-fidelity column makes this reviewable, but the verification itself is human-judgement.
- **Worst-offender files (using-habeebs-skill, prior-art-research at 13 hits each) may carry load-bearing chain-handoff citations.** Grill must pressure-test before Slice 4 applies cleanups to these.

## Vertical slices

Numbered in dependency order. Each slice cuts end-to-end. HITL = human-in-the-loop required; AFK = autonomous-friendly.

### Slice 1 — Audit script + per-file cleanup report (HITL: approval gate)

**Description:** Build a bash audit script that scans the 12 affected SKILL.md files for the three cruft regex patterns, generates a per-file change-list with before/after snippets and proposed actions, and writes the report to `docs/agents/specs/v1.21.0-body-cleanup-audit.md`. User reviews + approves the report before any cleanup commit lands. THIS IS THE LOAD-BEARING SLICE — user explicit instruction "audit first, then decide" gates every subsequent slice.

**Acceptance criteria:**
- [ ] Audit script at `skills/release/scripts/audit-skill-body-cruft.sh` (re-uses release skill's script home; not a new dir)
- [ ] Script scans all `skills/*/SKILL.md` files (not just the 12 known affected — must surface ANY hits across all 18 skills)
- [ ] Script detects 3 regex patterns: (a) `\bADR-\d{4}\b` in body prose (excluding frontmatter, excluding within `<!--` HTML comments); (b) `\b(Added|Introduced|Planned for|Phase \d+\.\d+ \(added)\s+v?\d+\.\d+\b` in body prose; (c) `\(documented \d{4}-\d{2}-\d{2}\)` and similar dated-incident patterns in body prose
- [ ] Script outputs per-file section in markdown report with: line number, before snippet (full line + 1 line context above/below), proposed action (REMOVE / RESTATE / MOVE-TO-POSTMORTEM / KEEP), restated-rule preview if RESTATE, postmortem target path if MOVE-TO-POSTMORTEM
- [ ] Report has summary table at top: file | hit count by category | proposed-action counts
- [ ] Report has `## Approval` section at bottom for user to sign-off per file (checkbox per file)
- [ ] Dogfood test at `tests/dogfood/26-audit-script-accuracy/check-audit-script.sh`: 3 fixture cases verifying (a) script catches a known ADR cite; (b) script catches a known version tag; (c) script does NOT trip on a frontmatter `metadata.version` field or an HTML comment
- [ ] Report regenerates idempotently (running script twice on same input yields identical report)
- [ ] User approval recorded in commit message of the audit-report commit (e.g., "Audit approved by Modie 2026-MM-DD; ready for Slice 2")

**Test strategy:** Integration test (bash dogfood) at `tests/dogfood/26-audit-script-accuracy/check-audit-script.sh` (3 fixture cases — happy / frontmatter-exclusion / HTML-comment-exclusion). Report itself is reviewed manually by user as the HITL gate.

**Blocked by:** None

**Notes:** This slice has TWO deliverables: (i) the script, (ii) the report. The script must run successfully and produce the report before user can approve. The user-approval step is OUT OF the agent's hands — the agent commits the script + report, the user reviews, the user signals approval (verbally + by allowing Slice 2 to proceed). No second commit needed in Slice 1 unless the audit reveals a script bug.
