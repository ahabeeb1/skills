## OQ-1 — Piece 3 PostToolUse validator scope (load-bearing; has kill-switch)

### Starting state

Four candidate warnings. Modie answered "not sure" at HITL pivot. Hook MUST justify existence or Slice 6 drops entirely.

### Axes grilled

**Failure modes (axis 2)** + **Reversibility (axis 6)** + **Observability (axis 7)** — these are the axes a hook lives or dies on.

### Pressure-test results

**(a) Missing spec when editing skills/ on a feature branch**
- *Failure mode probe:* Misfires on chore PRs that touch skills/ without a spec. Today's PR #47 didn't touch skills/ — so this specific example doesn't trigger — BUT release-skill itself touches skills/ at version-bump time, and chore PRs renaming files would all fire. False-positive rate: high enough to cause warn-fatigue within 2-3 PRs.
- *Observability:* The hook would print "warning: editing skills/foo/SKILL.md on feature branch without `docs/agents/specs/<slug>.md`" — but `<slug>` is unknown to the hook (would need to parse branch name? feature/foo → specs/foo.md?). Branch-naming convention is informal in this repo.
- *Verdict:* REJECT. False-positive prone; relies on branch-naming convention that doesn't exist.

**(b) Missing grill record when status: Grilled (drift detection)**
- *Failure mode probe:* Requires reading YAML frontmatter of `docs/agents/specs/*.md` files to find ones with `Status: Grilled` and then checking for a corresponding `<slug>-grill.md`. Is this state-owning? **Read-only check of file contents = stateless under ADR-0003** (silent-contradiction-3 resolution from research). Same shape as the existing peer-scan hook, which reads `.git/habeebs-sessions/*.json`. Not a violation.
- *Misfire probe:* Could a spec legitimately be in `Status: Grilled` without a grill record file? Currently no — but v1.21.0 spec inlined the grill amendment into the spec itself (`## 2026-05-26 Amendment`). If THIS run's grill record gets inlined into the spec post-grill, then a future spec with `Status: Grilled` and no separate grill file is legitimate. False-positive risk: moderate-low; depends on convention.
- *Value:* Catches a real drift class — spec marked Grilled but no record of what was grilled.
- *Verdict:* ACCEPT IF the convention "grill record lives in `<slug>-grill.md`" is enforced. v1.20.0 + v1.21.0 + this v1.22.0 grill all use `<slug>-grill.md` shape → convention holds 3/3 most recent. Adopt.

**(c) Editing on main/non-worktree path for non-trivial changes**
- *Failure mode probe:* Duplicates the existing `preventing-commits-to-default.sh` PreToolUse Bash block? Different layer — that hook blocks `git commit`; this would warn on `Edit|Write` BEFORE commit, when the change still has time to be moved to a worktree.
- *Misfire probe:* Defining "non-trivial" is hard without state. Hook can check: (i) on default branch? (ii) editing skills/, hooks/, or .claude-plugin/ paths? (iii) git status shows >0 uncommitted changes. All three signals are stateless.
- *Value:* Today this exact warning would have fired on the plan-status-flip edit in main checkout (the unstaged edit before chore PR #47 was created). Warning would have caught "editing on main, not in worktree, change is non-trivial" → "you might want to branch." Modie made the right call by hand; the hook would back it.
- *Verdict:* ACCEPT. Complements existing PreToolUse without duplication. Composes naturally.

**(d) Stale plan Status fields >2 releases**
- *Threshold probe:* What's "stale"? 2 releases? 3? Last 90 days? At Modie's solo cadence (~6 releases in May 2026), 2 releases = ~10 days. Too tight; would warn on every edit to recent plans during normal review.
- *Overlap probe:* Duplicates Piece 5's release-time editorial scan (which runs once per release, on the same data). PostToolUse would fire on every Edit; release scan fires once. Per-edit is wasteful; release-time is sufficient.
- *Verdict:* REJECT. Covered by Piece 5 release scan; per-edit redundancy would compound warn-fatigue.

### Resolution

**DECIDED — Piece 3 ships with scope = (b) + (c) only.**

- (b) missing grill record when spec `Status: Grilled` and no `<slug>-grill.md` exists
- (c) editing skills/ | hooks/ | .claude-plugin/ on default branch with uncommitted changes

(a) rejected (false-positive prone, branch-naming relies on absent convention).
(d) rejected (covered by Piece 5; redundant).

Hook stays warn-only, exit 0 always, stderr output. Honors `HABEEBS_DISABLE_HOOKS=1`.

**Kill-switch outcome:** Hook IS justified — does NOT drop. Slice 6 ships.

**Push back into spec:** Slice 6 acceptance criterion 4 ("Dogfood scenario 29 created with N fixture cases") → N = 4 (2 warning cases × 2 fixture pairs each: positive trigger + negative no-trigger control).

## OQ-2 — Piece 5 telemetry field naming

### Starting state

`last-reviewed` vs `last-updated`? `superseded-by` (kebab) vs `superseded_by` (snake)? Current ADR frontmatter is PascalCase (`Status:`, `Date:`, `Deciders:`, `Tier:`).

### Axes grilled

**Migration (axis 5)** + **Reversibility (axis 6)**.

### Pressure-test results

**Case sensitivity / convention drift:**
- Current convention is PascalCase Title-Case for ADR keys. New fields adopting lowercase-kebab would create within-file convention drift (mixed `Status:` and `last-reviewed:` in the same frontmatter block). Bad signal.
- Switching ALL keys to lowercase-kebab (`status:`, `date:`, `deciders:`, `tier:`, `last-reviewed:`, `superseded-by:`) is a mass-rename that v1.22.0 explicitly defers (Concrete pick #11 — new ADRs only).
- Compromise: NEW fields adopt PascalCase to match existing: `Date-Created:`, `Last-Reviewed:`, `Superseded-By:`. Zero convention drift; back-fill of existing ADRs becomes trivial-if-needed (just add 3 new keys; no key renames).
- *Verdict:* Adopt **PascalCase to match existing convention**.

**Semantic clarity — "last-reviewed" vs "last-updated":**
- "last-updated" auto-bumps on every commit. Defeats the dormancy signal — release-skill scan would find every ADR fresh after every release.
- "last-reviewed" implies deliberate review. Updated only when a human says "I reviewed this and confirm Status remains correct."
- Piece 5's release-skill scan needs deliberate-review semantics.
- *Verdict:* Adopt **`Last-Reviewed:`**.

**Resolution:**

```yaml
---
Status: <Proposed | Accepted | Active | Done | Deprecated | Superseded by ADR-N>
Date-Created: 2026-05-26    # never changes after initial write
Last-Reviewed: 2026-05-26   # deliberate-review timestamp; updated on Status transition OR explicit "still valid" review
Superseded-By: null         # path to replacement ADR/plan; null until superseded
---
```

### Resolution

**DECIDED.** Four fields, PascalCase to match existing convention. `Last-Reviewed:` semantics = deliberate review only, NOT auto-bump.

**Push back into spec:** Slice 4 acceptance criterion 1 ("four new fields in frontmatter") → use PascalCase exact field names above.

## OQ-3 — Piece 5 release-time editorial step cadence

### Starting state

Every release vs minor+major only?

### Axes grilled

**Observability (axis 7)** + **Failure modes (axis 2)**.

### Pressure-test results

**v1.21.0 chore PR #47 hypothetical:**
- PR #47 was a CHORE (post-release plan-status flip). Not a release PR. Release scan wouldn't have run on #47.
- The scan would have run on v1.21.0 RELEASE PR — at which point plan-v1.21.0 had `Status: Active` (correct for active work). Scan would flag nothing.
- So the scan would NOT have caught PR #47's need to exist. Scan is a different signal — it catches plans/ADRs that go stale across releases, not within-release activity.

**Patch release noise probe:**
- Patches in habeebs-skill are wording fixes (v1.18.0 → v1.18.1 chore release pattern). Patch releases typically don't touch ADR or plan content.
- IF a patch release scan ran, it would re-flag the SAME stale items the minor release flagged. Redundant noise — same finding, twice.
- BUT patches are infrequent (1-2 per year). Cost is low.

**Threshold pressure:**
- Spec says "last-reviewed > 3 releases ago." At Modie's cadence (~6 releases in May 2026 alone), 3 releases ≈ 2 weeks. Probably too tight — most ADRs won't be reviewed every 2 weeks. False-positive prone.
- Better threshold: 6 months OR 3 minor releases (whichever comes first). At today's cadence: 3 minor ≈ 6 weeks. Still tight but actionable.

### Resolution

**DECIDED — minor+major only, threshold 3 minor releases OR 6 months (whichever first).**

- Patch releases skip the editorial scan (low value, redundant with most-recent minor scan).
- Threshold: `Last-Reviewed:` older than max(3 minor releases ago, 6 months ago).
- The hybrid is honest about Modie's variable cadence — 3 minors might be 6 weeks OR 6 months depending on release activity. 6 months is the cadence-independent floor.

**Push back into spec:** Slice 4 acceptance criterion 4 ("release/SKILL.md Phase 10 has editorial-scan step") → spec the threshold as `max(3-minor-releases-ago, 6-months-ago)` and run on minor+major only.

## OQ-4 — Piece 1 jargon glossary placement

### Starting state

Inline-at-bottom vs extend GLOSSARY.md. Pre-flight grep: GLOSSARY DEFINES `slice`, `HITL slice`, `AFK slice` (lines 32, 35-36). DOES NOT define `pgroup`, `pre-flight verification`, `approval-gate`, `tracer slice`, `AFK:full-auto`, `HITL:per-file`.

### Axes grilled

**Migration (axis 5)** + DX (devex-review extension).

### Devex-review contributions

`devex-review` fired (Piece 1 is contributor-facing). Surfaced questions:
- **Onboarding friction:** A new contributor reading a plan with `pgroup-2A blocked by #1, #6, README-verify` hits 3 undefined terms in one cell. Inline-at-bottom-of-plan solves this for ONE plan; GLOSSARY.md solves it across all plans. GLOSSARY wins for cross-plan terms.
- **Upgrade friction:** Cross-plan terms (`pgroup`, `HITL:per-file`) change rarely. Per-plan terms (`pre-Slice-2 verification`) are ad-hoc and don't belong in GLOSSARY. Different lifecycle.

### Pressure-test results

**Cross-plan vs ad-hoc test:**
- `pgroup`, `HITL:approval-gate`, `HITL:per-file`, `AFK:full-auto`, `tracer slice`, `pre-flight verification` — these recur across plans. Cross-plan.
- `pre-Slice-2 verification`, `pgroup-2A`, `(dogfood scenario #28)` — these are plan-specific identifiers. Ad-hoc.

**Verdict:** Different categories, different placement. Cross-plan terms go to GLOSSARY (one canonical definition). Ad-hoc plan-specific identifiers stay inline (in-context).

**The 5 undefined cross-plan terms:**
1. `pgroup` (parallel-group; dispatch group from parallel-dev)
2. `pre-flight verification` (Slice 0 / Slice-0.5 readiness check)
3. `HITL:approval-gate` / `HITL:per-file` / `HITL:inline` (subtypes of HITL slice)
4. `AFK:full-auto` (the AFK extreme — no human gating at all)
5. `tracer slice` (lowest-risk first slice that surfaces format/coupling problems)

Adding these 5 to GLOSSARY.md is a one-commit add — low cost, high value. In scope of Slice 3 (Piece 1 plan format work) since they're the terms the new plan format will use.

### Resolution

**DECIDED — Cross-plan jargon → GLOSSARY.md (extended in Slice 3); ad-hoc plan identifiers → inline in the plan that uses them.**

- Slice 3 deliverable expands to include adding the 5 missing entries to GLOSSARY.md (per ADR-0005 single-writer invariant — GLOSSARY is human-authored; this is a human-driven add via the spec).
- Plan template (in references/plan-template.md) instructs writers: ad-hoc identifiers stay inline (`pre-Slice-2 verification = ...`); cross-plan terms stay undefined inline and rely on the GLOSSARY link in the doc footer.

**Push back into spec:** Slice 3 acceptance criteria gain a new bullet — "GLOSSARY.md has entries for the 5 cross-plan jargon terms listed in the grill record."

## OQ-5 — Piece 4 cross-reference cleanup scope

### Starting state

Spec said: grep first, then decide. Threshold: ≤5 = simple; 6-20 = larger-than-expected; >20 = re-scope or split.

### Grep findings (load-bearing)

28 hits across 10 active-surface files. **But the hit-count framing was wrong.**

Reading the actual content of the hits revealed:
- `skills/cross-session-detect/audit.sh` (2 hits): **LIVE RUNTIME WRITER** — writes conflict JSON files to `docs/agents/conflicts/<id>.json` at line 41
- `tests/audit/audit_writer_test.sh` (4 hits): tests that depend on `docs/agents/conflicts/` as the writer path
- `tests/actions/actions_test.sh` (8 hits): same — tests the cross-session-detect actions that write to `conflicts/`
- `skills/parallel-dev/SKILL.md` (1 hit, line 191): documents `docs/agents/dispatches/<dispatch-id>.json` as the writer path
- `skills/parallel-dev/references/dispatch-record-template.md` (3 hits): documents same
- `skills/tdd-loop/SKILL.md` (2 hits, lines 89 + 111): references `docs/agents/dispatches/` for dispatch record reading + DONE_WITH_CONCERNS append
- `tests/dogfood/10-pgroup-dispatch/10a + 10b` (3 + 3 hits): dogfood scenarios for pgroup dispatch — reference `dispatches/`
- `CLAUDE.md` (1 hit, line 85): inventories `dispatches/` as a tracked surface
- `AGENTS.md` (1 hit, line 40): same

**Both directories are RUNTIME WRITER TARGETS, not dormant scaffolding.**

- `dispatches/` is where `parallel-dev` Phase 7.5 writes dispatch records (declared by ADR-0018 Part A, writer implemented in v1.17.0)
- `conflicts/` is where `cross-session-detect/audit.sh` writes conflict audit JSON at runtime (declared by ADR-0019, writer implemented in v1.18.0)

Both have `.gitkeep` files. They look empty in git snapshots because:
- `dispatches/` requires `parallel-dev` Phase 7.5 to have run AND for the user to have committed the resulting JSON. Parallel-dev was used during research (this run) but the resulting JSON sits in conversation/transcript, not on disk yet.
- `conflicts/` requires an actual cross-session conflict to have been detected. Modie hasn't hit one in production.

### This invalidates Piece 4's framing

The research § Piece 4 + ADR-0021's original cut treated these as "dormant" — but they're **inherently sparse** writer targets, not dormant. MADR's "remove the empty optional section" doesn't apply because these are NOT empty optional sections — they're functional writer paths. Closer analog: a `logs/` directory that has 0 entries after `rm logs/*.log` doesn't get deleted; it stays for the next write.

The research synthesis missed this because SP4 (Dormant directory retirement) was scoped to "ADR-declared-but-empty" abstractly. It didn't probe the **writer-path** dimension. Silent-contradiction-2 from pattern-extractor applies at the wrong scope.

### Resolution

**DECIDED — Piece 4 restructured.** Surfaces OQ-6 below.

OQ-5 itself is now moot — there are no "dispatches/conflicts cross-references to clean" because those references are **load-bearing writer paths**, not dead links. They stay.

**Push back into spec:** Slice 1 description rewritten in spec amendment below. The grep step still runs in Slice 1 — but produces a DIFFERENT outcome (verifying writer paths are intact, not deleting references).

## OQ-6 — Piece 4 restructured (surfaced during OQ-5 grill)

### Starting state

Original Piece 4: delete `docs/agents/dispatches/` and `docs/agents/conflicts/`; append re-amendment block to ADR-0021. Per OQ-5 finding, this would break v1.17.0 and v1.18.0 runtime writers.

### Axes grilled

**Failure modes (axis 2)** + **Migration (axis 5)** + **Reversibility (axis 6)**.

### Reframing

Two distinct sub-problems were conflated by the original research synthesis:

**A. Are dispatches/ + conflicts/ "dormant scaffolding"?**

NO. They're runtime writer targets. They look empty because:
- `dispatches/` writers haven't run in production (parallel-dev Phase 7.5 is used during research; this run's pattern-extractor + source-fetchers should have written records if Phase 7.5 fired — they didn't, because Phase 7.5 is partly dormant-as-mechanism even though the dir is well-defined)
- `conflicts/` writers haven't run because no conflicts have been detected in production

The right action is NOT to delete the dirs. The right action is one of:
1. **Keep dirs with `.gitkeep`; document the writer-path contract explicitly** in CLAUDE.md + AGENTS.md so the "is this empty dir dormant?" question doesn't recur. CURRENT STATE; just add prose.
2. **Move both dirs out of git entirely** — make them runtime artifacts gitignored from `.gitignore`. Loses the visibility that the writer paths exist as conventions. Worse.
3. **Promote runtime-state directories to a separate convention surface** like `.habeebs/runtime/dispatches/` + `.habeebs/runtime/conflicts/` — gitignored, with .keep files inside `docs/agents/` describing where to look. Substrate-coupled — violates ADR-0002 spirit.

(1) is the simplest correct answer.

**B. Is ADR-0021's 2026-05-25 amendment correct?**

YES, BUT under-specified. The amendment says "carved out from Pattern G" but doesn't articulate WHY (writer-path semantics). v1.22.0 should land a clarification, not a reversal.

### Resolution

**DECIDED — Piece 4 restructured to "Clarify ADR-0021 carve-out semantics" instead of "Delete + re-amend."**

**New Piece 4:**
1. Append "## 2026-05-26 Clarification" block to ADR-0021 (NOT a re-amendment — a clarification that the 2026-05-25 amendment was correct in spirit but under-articulated). Block content articulates:
   - `dispatches/` and `conflicts/` are RUNTIME WRITER TARGETS (declared by ADR-0004 Part 2 + ADR-0018 Part A + ADR-0019)
   - Their on-disk emptiness in git snapshots reflects RUNTIME-PATH SPARSITY, not dormancy
   - Pattern G's "earn existence by file count" applies to AUTHORED methodology directories (`adrs/`, `specs/`, `plans/`), NOT to RUNTIME WRITER PATHS
   - Future rule: any ADR declaring a runtime writer path MUST land the writer implementation in the same release; absence-of-writer means absence-of-directory
2. Add ONE prose paragraph to CLAUDE.md § "Agent skills" inventory describing the writer-path category vs the authored-methodology category
3. Add ONE prose paragraph to AGENTS.md mirror
4. NO directory deletion. NO file cleanup of the 28 cross-references — those are all load-bearing pointers to live writer paths.
5. ADR-0021 Status line gains a third element: `Accepted (amended 2026-05-25, clarified 2026-05-26)`

This is a SMALLER slice than the original Piece 4. Estimated 1 commit, 1 hour.

**Push back into spec:** Slice 1 description fully rewritten — see `## 2026-05-26 Amendment` to spec below.
