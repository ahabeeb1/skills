## Per-item resolutions

### Item 1 — Q1: 4-status contract verbatim vs fork

- **Starting state:** Spec proposed verbatim adoption from Superpowers `subagent-driven-development/implementer-prompt.md` but flagged drift risk
- **Axes grilled:** Reversibility, Migration
- **Key questions:** What's the actual drift cost? Does any habeebs case fail the four canonical statuses? What's the cost of forking later if needed?
- **Resolution:** DECIDED
- **Decision:** **Adopt verbatim.** ADR-0004 captures snapshot semantics as of 2026-05-12. Our copy is canonical for habeebs-skill regardless of Superpowers' future changes. Five-whys probe: every candidate habeebs-specific addition (e.g., `DONE_REVIEW_PENDING`) collapses to `DONE_WITH_CONCERNS` in practice. Adding a 5th status later is trivial; removing one is breaking.
- **Revisit trigger:** A real-world dispatched-subagent case that doesn't fit any of the four statuses → propose 5th status with ADR amendment
- **Spec update:** None — spec already specifies verbatim

### Item 2 — Q2: Dispatch state file location

- **Starting state:** Three candidates: `docs/agents/dispatches/<id>.json` (centralized), `.dispatches/` gitignored, per-worktree state file
- **Axes grilled:** Reversibility, Observability, Migration
- **Key questions:** Who reads the dispatch record during chain execution? Who reads it post-hoc? What audit story do we lose if records aren't tracked?
- **Resolution:** DECIDED
- **Decision:** **`docs/agents/dispatches/<dispatch-id>.json`** (centralized, in-repo, gitignored OFF). Single-writer invariant preserved: only dispatcher writes after subagents return SHAs. No skill reads records during chain execution (audit-only). The `.dispatches/` gitignored variant fails observability — `git log --grep "Dispatch-id:"` from `parallel-dev` SKILL.md becomes the secondary index. Per-worktree fragmentation fails F5 (state unification).
- **Revisit trigger:** Directory exceeds 1000 records → introduce retention policy with `/sync` cleanup pass
- **Spec update:** Slice 6 acceptance criterion: add `docs/agents/dispatches/.gitkeep`; explicitly note "write-only by dispatcher, no in-execution reads"

### Item 3 — Q3: Concurrency cap

- **Starting state:** Hardcoded 5 vs configurable per-dispatch
- **Axes grilled:** Performance, Reversibility
- **Key questions:** When does 5 bind? Is the override surface coherent if added later?
- **Resolution:** DECIDED
- **Decision:** **Default soft-cap of 5 with optional per-pgroup override** via `concurrency: <N>` field in pgroup labeling. Most pgroups won't override. Removing the override field later is breaking; adding it now is opt-in (silent default = 5). Source: appxlab's empirical 5-7 ceiling.
- **Revisit trigger:** Median override usage >50% of pgroups → rethink default; or Claude Code v2.2+ raises the practical ceiling
- **Spec update:** Slice 4 acceptance: document `concurrency: <N>` optional field in `parallel-dev/SKILL.md`; Slice 6 dispatch-record-template includes `concurrency_used` field

### Item 4 — Q4: Re-fan-out cycle budget for Phase 2.5 critic

- **Starting state:** Research suggested 1, LangGraph default is 2
- **Axes grilled:** Failure modes, Performance
- **Key questions:** What's the failure mode of 0 cycles? Of 2 cycles? Where does the lead's judgment sit in the loop?
- **Resolution:** DECIDED
- **Decision:** **Hard-stop at 1 iteration**, no auto re-fan-out in v1.7.0. The critic produces a list of missing categories; lead decides accept/reject. If lead rejects without justification, the "Phase 2.5 outcome" section in `output-template.md` requires written reason per rejection (R1/R2 mitigation).
- **Revisit trigger:** Phase 6 CitationAgent (v1.8.0+) ships → reconsider auto re-fan-out at Phase 2.5; or dogfood scenarios 07a/b/c reveal that 1 iteration consistently misses real gaps
- **Spec update:** Slice 5 acceptance: add "Phase 2.5 outcome" section requirement to `output-template.md` update

### Item 5 — Q5: ADR-0002 amendment

- **Starting state:** Is dispatch record an "audit log" (compliant with ADR-0002) or a "state substrate" (violating)?
- **Axes grilled:** Reversibility, Concurrency, Migration
- **Key questions:** Does any skill read the file during execution? Does it require locking? Is it a daemon? Is it queue-like?
- **Resolution:** DECIDED
- **Decision:** **ADR-0004 carves out dispatch records as a permitted in-repo artifact category; ADR-0002 body untouched.** Substrate test fails on all 4 criteria: no daemon, no IPC, no concurrent access (single writer), no in-flight reads. ADR-0002's "no runtime substrate" rule targeted runtime daemons / queues / locks — not static JSON written once per dispatch. ADR-0004 references ADR-0002 so future readers find the rationale.
- **Revisit trigger:** Any skill starts reading dispatch records mid-chain → substrate violation, requires ADR-0002 amendment
- **Spec update:** None — Slice 6 already names the location; ADR-0004 formalizes

### Item 6 — Q6: Phase 2.5 critic dogfood acceptance test rigor

- **Starting state:** Single planted-incomplete-decomposition scenario at `tests/dogfood/07-category-critic.md`
- **Axes grilled:** Observability, Failure modes
- **Key questions:** What if the critic memorizes one pattern? How do we prove it generalizes? How do we catch false positives (hallucinated categories)?
- **Resolution:** DECIDED
- **Decision:** **Multi-scenario adversarial suite at `tests/dogfood/07-category-critic/` (directory):**
  - **07a — missing observability:** planner sees "build a job queue", misses "alerting/metrics/dashboards". Critic must surface; lead must accept.
  - **07b — missing hooks** (the literal v1.6.0 miss): planner sees "plugin methodology", misses "event handlers/hooks". Critic must surface; lead must accept.
  - **07c — missing security:** planner sees "user uploads", misses "auth/authz/file validation". Critic must surface; lead must accept.
  - **07d — no-gap control:** planner produces a complete decomposition. Critic must return `{approved: true}` (no hallucinated additions). This is the false-positive gate.
- **Revisit trigger:** Real-world critic miss in v1.7.x → add scenario to suite; any 07d-style hallucination → tune critic prompt
- **Spec update:** Slice 5 acceptance criterion expanded — 4 scenarios (a/b/c/d), not 1; rename `07-category-critic.md` → `07-category-critic/` directory

### Item 7 — Q7: Hypothesis-probe sub-pattern home

- **Starting state:** `parallel-dev` (dispatcher) vs `systematic-debugging` (consumer)
- **Axes grilled:** Reversibility, F10 (focused agents)
- **Key questions:** Where do consumers expect to find the pattern? Where does the cross-ref point?
- **Resolution:** DECIDED
- **Decision:** **`parallel-dev` documents the sub-pattern in v1.7.0; `systematic-debugging` cross-refs it when consumed in v1.8.0+.** Dispatcher owns dispatch patterns; consumer cross-refs. F10 honored (both skills stay focused).
- **Revisit trigger:** None — clean separation
- **Spec update:** None — Slice 4 acceptance already names `parallel-dev` as the home

### Item 8 — Q8: Pgroup detection — entry-time vs recursive

- **Starting state:** Flat list (entry-time) vs DAG (recursive: completed pgroups unblock new pgroups)
- **Axes grilled:** Concurrency, Migration, Performance
- **Key questions:** What does `write-plan`'s output shape need to change to support recursive? What's the dogfood gain?
- **Resolution:** DECIDED
- **Decision:** **Entry-time only in v1.7.0.** Recursive requires `write-plan` output schema change (pgroups as DAG, not flat list) — that's a structural v1.8.0+ change. Flat-list works for dogfood scenario 08 and for the spec's own pgroup-1A/pgroup-2A layout. Performance cost is negligible at typical plan sizes.
- **Revisit trigger:** A real plan emerges where flat pgroups force avoidable sequentialization (e.g., a 12-slice plan with a complex unblocking graph) → introduce recursive detection
- **Spec update:** Slice 6 acceptance: explicitly note "entry-time pgroup walk; no re-detection on slice completion"

### Item 9 — AF-F6: Pause/resume on subagent dispatch interruption

- **Starting state:** Spec silent
- **Axes grilled:** Failure modes, Reversibility, Concurrency
- **Key questions:** What persists across interruption? How does re-invocation know what's done vs pending? Where's the durability layer?
- **Resolution:** DECIDED
- **Decision:** **No explicit pause/resume API in v1.7.0. Git is the durability layer.** Idempotent re-invocation: user re-runs `tdd-loop`; it inspects existing commits via `git log --grep "Dispatch-id: <id>"` and existing branches/worktrees, skips completed slices, re-dispatches pending ones. Subagent's incomplete in-flight work (uncommitted) is lost on kill — same as any interrupted TDD cycle; user resolves manually. F6's "simple API" requirement satisfied by idempotent re-invocation.
- **Revisit trigger:** Real-world case where re-invocation produces incorrect results (e.g., a slice was "almost done" with uncommitted work that gets silently discarded) → introduce explicit checkpoint mechanism
- **Spec update:** Slice 6 acceptance: add "tdd-loop Phase 0.5 detects existing completed slices via git history before dispatching" requirement

### Item 10 — AF-F7: Human escalation surface for BLOCKED

- **Starting state:** Spec implies free-form text surfaces to user
- **Axes grilled:** Failure modes, F7 (human-as-tool)
- **Key questions:** What's the message shape? What's the user's expected next action? Is this a tool call or a chat message?
- **Resolution:** DECIDED
- **Decision:** **Structured BLOCKED output** in `tdd-loop` user-facing emission. Shape:
  ```
  {
    type: "BLOCKED",
    subagent: <name>,
    slice_id: <#N>,
    reason: <one-line description>,
    suggested_action: "edit-spec-and-redispatch" | "investigate-manually" | "escalate-to-maintainer"
  }
  ```
  This is part of the 4-status return contract spec in Slice 4. No async approval queue (out of scope for v1.7.0); synchronous user gate is the only escalation surface.
- **Revisit trigger:** 3+ user reports that BLOCKED messages lack actionable context → enrich schema (e.g., add `diagnostic_logs`, `next_attempt_hint`)
- **Spec update:** Slice 4 acceptance: add BLOCKED message schema to the 4-status return contract section

### Item 11 — AF-F1 / F4: Subagent call input + return JSON schemas

- **Starting state:** Spec mentions 4-status return but no input schema
- **Axes grilled:** Failure modes, F4 (structured outputs)
- **Key questions:** Is the input contract complete? Is the return parseable without ambiguity?
- **Resolution:** DECIDED
- **Decision:** **Both schemas captured in `skills/parallel-dev/references/dispatch-record-template.md` as part of Slice 4.**

  **Input schema:**
  ```json
  {
    "dispatch_id": "string",
    "subagent_name": "string",
    "task": "string (1-3 sentences)",
    "input_files": ["path"],
    "output_path": "path",
    "constraints": ["string"],
    "verification": "string",
    "worktree_path": "path",
    "branch": "string",
    "context_preamble": "string (SYSTEM_CONTEXT.md content; see AF-F13)"
  }
  ```

  **Return schema:**
  ```json
  {
    "status": "DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT",
    "commit_shas": ["sha"],
    "duration_ms": "int",
    "total_tokens": "int",
    "notes": "string (free-form; required when status = DONE_WITH_CONCERNS)",
    "blocker": "string (required when status = BLOCKED; matches AF-F7 schema)",
    "context_request": "string (required when status = NEEDS_CONTEXT)"
  }
  ```
- **Revisit trigger:** A real subagent output that doesn't fit the schema → propose schema amendment
- **Spec update:** Slice 4 acceptance: schemas added to `dispatch-record-template.md`

### Item 12 — AF-F13: Pre-fetch SYSTEM_CONTEXT.md into subagent prompts

- **Starting state:** Spec silent
- **Axes grilled:** Performance, F3 (own context window)
- **Key questions:** What's the cost of injecting SYSTEM_CONTEXT.md into each subagent? Of NOT injecting it? Where does it live in the input schema?
- **Resolution:** DECIDED
- **Decision:** **Yes — dispatcher pre-fetches `SYSTEM_CONTEXT.md` content and injects as `context_preamble` in each subagent's input** (per Item 11 schema). Without this, subagents may re-run Phase 0 reconnaissance independently — token waste + drift risk. Single-writer invariant preserved (subagents read, dispatcher's parent already wrote). Token cost of 1-2KB preamble dwarfed by re-recon cost.
- **Revisit trigger:** `SYSTEM_CONTEXT.md` grows past ~5KB → consider injecting a derived summary instead of full content
- **Spec update:** Slice 4 acceptance: add "subagent prompt preamble: `SYSTEM_CONTEXT.md` content prepended as `context_preamble` field" requirement

### Item 13 — AF-F9: DONE_WITH_CONCERNS handling

- **Starting state:** Spec says "log concerns inline" — vague
- **Axes grilled:** Observability, F9 (compact errors)
- **Key questions:** What does "log" mean — file, terminal, both? Is this a warning, alert, or blocker?
- **Resolution:** DECIDED
- **Decision:** **Notes field appended to dispatch record + emitted as warning in `tdd-loop` output** (visible but non-blocking). Not an alert, not a halt — just visible enough that the user sees concerns when reviewing the chain output.
- **Revisit trigger:** Users report ignoring DONE_WITH_CONCERNS warnings systematically → consider elevating to halt-with-confirm
- **Spec update:** Slice 6 acceptance: clarify "DONE_WITH_CONCERNS → advance + emit warning in tdd-loop output + append notes to dispatch record"

### Item 14 — AF-F11: New trigger surfaces beyond `tdd-loop` / `prior-art-research`

- **Starting state:** `parallel-dev/SKILL.md` already lists user-direct `/parallel` trigger
- **Axes grilled:** Reversibility (low blast radius — trigger surface)
- **Key questions:** Are we adding any new trigger surface in v1.7.0?
- **Resolution:** DECIDED
- **Decision:** **No new trigger surfaces in v1.7.0.** Existing triggers (`prior-art-research` Deep mode, `tdd-loop` Phase 0.5 once Slice 6 ships, `/parallel` user-direct) cover all v1.7.0 needs. F11 says trigger from anywhere — at v1.7.0 maturity, "anywhere" = these three; broader integration (Slack, cron, etc.) is out of scope.
- **Revisit trigger:** Real demand for new trigger surface (e.g., a webhook from CI to dispatch a fix-the-flaky-test pgroup) → add as v1.8.0+ feature
- **Spec update:** None

## New decisions surfaced during grilling

These weren't in the spec's Open Questions but emerged during the grill:

1. **Subagent input schema is a load-bearing artifact, not just a hint.** Item 11. Becomes a binding contract in Slice 4 dispatch-record-template.
2. **`SYSTEM_CONTEXT.md` preamble injection is mandatory** (Item 12), not optional. Becomes a required field in the input schema.
3. **BLOCKED message gets a structured shape with `suggested_action` enum** (Item 10). User UX surface, not just internal.
4. **Idempotent re-invocation is the pause/resume API** (Item 9). Required: tdd-loop Phase 0.5 inspects existing git history before dispatching. This is a *new* requirement on Slice 6 that wasn't there before.
5. **Dogfood scenario 07 expands from one file to four scenarios in a directory** (Item 6). Critical for R2 mitigation. Slice 5 acceptance updated.
6. **`concurrency: <N>` opt-in override field on pgroup labeling** (Item 3). New surface; backward-compatible default.
