## Per-item resolutions

### Item 1 — Harness compatibility

- **Starting state:** Spec defaulted to "no, but verify with quick check before commit."
- **Axes grilled:** Migration, Failure modes.
- **Key questions:**
  - Does any harness reserve `docs/agents/*.md` as a special path?
  - What's the failure mode if one does?
- **Findings:** Codex uses `AGENTS.md` at repo root; Cursor uses `.cursor/rules/` + `.cursorrules`; OpenCode uses `.opencode/`; Claude Code uses `CLAUDE.md` + `.claude/`. None reserve `docs/agents/`. The directory is a habeebs-internal convention.
- **Resolution:** **DECIDED.**
- **Decision:** Rename safe. No harness collision.
- **Revisit trigger:** A future harness adds a `docs/agents/*.md` convention — re-evaluate (trigger already in spec § Revisit triggers).
- **Spec update:** None.

### Item 2 — Phase 0 failure handling

- **Starting state:** Spec defaulted to "warn-only + retry hint" (no rollback of GLOSSARY/tracker/labels).
- **Axes grilled:** Failure modes, Observability, Reversibility.
- **Key questions:**
  - What are Phase 0's actual failure surfaces? (Filesystem-only; no network.)
  - Could the user assume setup succeeded but later hit halt-on-missing-SYSTEM_CONTEXT?
  - Are write-failures (rare/hard) and `[unknown]` tags (common/soft) the same failure mode?
- **Findings:** The original "warn-only" framing conflated two distinct scenarios. Write-failures are rare and hard-to-miss; `[unknown]` tags are Phase 0's normal soft output, not failures at all.
- **Resolution:** **DECIDED.**
- **Decision:** Fork by failure type:
  - **Write failure** (permission denied, disk full, sandbox restriction, git uninitialized): halt-loud at end of setup with `SETUP_INCOMPLETE` banner + recovery command. Existing GLOSSARY/tracker/labels preserved.
  - **`[unknown]` tags**: pass-through. Setup confirms success; prints count of `[unknown]` fields.
- **Spec update:** Slice 2 Phase 7 failure-handling clause rewritten (applied).
