## Per-item resolutions

### Item 1 — "Inspired by X" credit placement

- **Starting state:** Spec says move credits out of frontmatter description but didn't specify body location.
- **Axes grilled:** Reversibility (trivial — can move later), Observability (none material).
- **Key questions:** Top of body or bottom? Single canonical section name across all skills? Distinguish "inspired by" (loose) from "lifted from" (direct)?
- **Resolution:** DECIDED.
- **Decision:** Add `## Origins` section near the bottom of each SKILL.md body (after `## See also`, before final `HANDOFF` block if present). Single canonical section name across all 14 skills. Use phrasing "Inspired by X" for loose influence (parallel evolution) and "Lifted from X" for direct borrowing (e.g., Ousterhout deep-modules, Pocock vertical-slice). Bottom placement keeps the trigger-relevant body content on top.
- **Spec update:** Slice 1 acceptance criteria gain: "Every SKILL.md with prior credits has an `## Origins` section near the bottom; phrasing uses 'Inspired by' or 'Lifted from' explicitly."

### Item 2 — Load-bearing "Do NOT use" lists

- **Starting state:** Spec asks whether some descriptions' anti-triggers are load-bearing for trigger discipline and shouldn't be trimmed.
- **Axes grilled:** Failure modes (false-positive triggers — over-triggering is the failure mode).
- **Key questions:** Which skills are high-blast-radius if they over-trigger? Which "Do NOT use" lines have empirically caught misuse?
- **Resolution:** DECIDED.
- **Decision:** Three skills retain a multi-bullet "Do NOT use" list because their pushy trigger language has the widest catchment and would otherwise over-fire: `prior-art-research`, `socratic-grill`, `tdd-loop`. For these, keep 2-3 anti-trigger clauses (the highest-precision ones). For the other 11 skills, condense to ≤1 line or remove if redundant with the capability statement. Concrete: `using-worktrees`, `parallel-dev`, `vertical-slice`, `decision-record`, `draft-spec`, `agent-factors-check`, `deep-modules`, `setup-habeebs-skill`, `systematic-debugging`, `write-plan`, `using-habeebs-skill` get condensed anti-triggers.
- **Spec update:** Slice 1 acceptance criteria gain: "Three keystone skills (prior-art-research, socratic-grill, tdd-loop) retain ≥2 anti-trigger bullets; remaining 11 condense to ≤1 line or removal."

### Item 3 — Chain-relationship survival post-`next-skills`-removal

- **Starting state:** Spec asked whether each removed relationship survives via HANDOFF line, `## See also`, or prose mention.
- **Axes grilled:** Failure modes (chain integrity breaks if relationship lost), Observability (how do we detect breakage).
- **Key questions:** Single canonical home for the relationship, or pick best per-skill? How is verification automated?
- **Resolution:** DECIDED.
- **Decision:** For each removed `(source_skill, target_skill)` pair, the relationship MUST appear in at least one of: (a) `HANDOFF:` line at the end of source SKILL.md body, (b) `## See also` body section listing target with one-line context, (c) explicit step-text mention by name. Most relationships already exist via HANDOFF (research → spec/grill/record; spec → grill/record/tdd; tdd → deep-modules/parallel-dev/systematic-debugging). Where neither HANDOFF nor prose covers it, add `## See also`. Slice 2 builds an assertion script (see item 12) to verify.
- **Spec update:** Slice 2 acceptance criteria add: "For each removed `(source, target)` pair, target name appears in source SKILL.md body via HANDOFF, `## See also`, or step-text prose. Verification script lands in `tests/dogfood/10-description-budget/`."

### Item 4 — verify-output GATE vs ANNOTATE; deep-modules call

- **Starting state:** Default behavior + relationship to deep-modules under-specified.
- **Axes grilled:** Reversibility (default UX is a one-way door — changing it later breaks user expectations), Failure modes (false-positive slop blocks).
- **Key questions:** What does the user see when slop is found? Is the user blocked, or just informed? Does deep-modules cover this already?
- **Resolution:** DECIDED.
- **Decision:** Default mode is ANNOTATE. The skill emits `DONE` (clean) or `DONE_WITH_CONCERNS` (slop listed) by default. `BLOCKED` returns ONLY when severe slop is found — defined as: half-finished implementations (functions with `TODO`/`pass`/`throw new NotImplementedError`), or unreachable code, or imports/variables declared and never used in the staged diff. Moderate slop (unjustified comments, defensive validation past system boundaries) returns `DONE_WITH_CONCERNS` only — never blocks. GATE mode is opt-in via skill arg `verify-output --gate` and is documented but NOT the default. Does NOT call `deep-modules` internally — that's a separate, deeper refactor concern already invoked at tdd-loop's refactor step. `verify-output` is a final-pass linter, narrower scope.
- **Spec update:** Slice 3 acceptance criteria add: "Default mode is ANNOTATE. `BLOCKED` returns only on severe slop (half-finished impl, unreachable code, declared-and-unused). GATE mode is opt-in via `--gate` arg. Does not invoke deep-modules internally."

### Item 5 — Slop heuristic source

- **Starting state:** Spec asked whether to lift CLAUDE.md rules verbatim + paraphrase OMC's `ai-slop-cleaner`.
- **Axes grilled:** Reversibility (heuristic can evolve via revisions to `references/slop-heuristics.md`), Failure modes (false-positive flags).
- **Key questions:** Which CLAUDE.md rules are slop-relevant vs scope-relevant? How many OMC heuristics, and what's the line on paraphrase vs copy?
- **Resolution:** DECIDED.
- **Decision:** `references/slop-heuristics.md` lifts the following from `CLAUDE.md` verbatim (it's user-authored canon for this repo, no copyright concern): "Don't add features, refactor, or introduce abstractions beyond what the task requires"; "Don't add error handling, fallbacks, or validation for scenarios that can't happen"; "Default to writing no comments. Only add one when the WHY is non-obvious"; "Avoid backwards-compatibility hacks". Adds 3 paraphrased OMC-inspired heuristics (no direct text copy, max 10 words quoted per source): (a) repeated boilerplate suggesting a missed abstraction, (b) defensive validation past trusted boundaries, (c) backward-compat shims for unshipped code. Each heuristic ships with a 2-sentence positive example + counter-example.
- **Spec update:** Slice 3 acceptance criteria add: "`references/slop-heuristics.md` lifts 4 CLAUDE.md rules verbatim + 3 OMC-paraphrased heuristics; each carries example + counter-example."

### Item 6 — /abort-chain cleanup state

- **Starting state:** Spec listed 2 cleanup steps (steering flush, worktree teardown) and asked "anything else?"
- **Axes grilled:** Failure modes (incomplete cleanup leaves stale state on next chain run), Reversibility (cleanup is mostly irreversible — chosen direction matters).
- **Key questions:** What in-flight artifacts can a chain leave behind? Does the cleanup do any git operations? How does the user confirm cleanup succeeded?
- **Resolution:** DECIDED.
- **Decision:** Cleanup checklist in this order:
  1. Flush `docs/agents/SYSTEM_CONTEXT.md` `## Active steering` block; copy to `## Last reconciliation outcome` with `(chain aborted YYYY-MM-DD — reason: <user-provided-or-inferred>)` header line.
  2. List in-flight worktrees via `git worktree list`; for each that matches the active chain's branch pattern, prompt user "Abandon worktree X? (y/n)"; on yes, run `using-worktrees` Phase 6 teardown (`git worktree remove` — only `--force` if explicitly confirmed a second time).
  3. If a partial spec exists in `docs/agents/specs/` younger than the chain start: move to `docs/agents/specs/abandoned/<original-name>.md` and prepend `**ABANDONED:** YYYY-MM-DD — <reason>` header.
  4. If a partial ADR exists in `docs/agents/adrs/` with status `Draft`: same archive move to `docs/agents/adrs/abandoned/`.
  5. Echo a final summary listing what was cleaned + what was preserved.
  - NO automatic destructive git operations beyond worktree teardown (no force-push, no branch deletion, no `git reset --hard`).
- **Spec update:** Slice 4 acceptance criteria expand from 4 items to 6 — the new section covers all 5 cleanup steps + the "no destructive git" rule.

### Item 7 — GLOSSARY LINK vs LOAD

- **Starting state:** Spec asked whether Phase 0 should LOAD GLOSSARY (always include in context) or LINK (read on demand).
- **Axes grilled:** Performance (token cost), Failure modes (term ambiguity slips through if LINK and agent doesn't bother reading).
- **Key questions:** What's the token cost of LOAD? How often does an agent need GLOSSARY content vs just need to know it exists? What's the empirical evidence?
- **Resolution:** DECIDED.
- **Decision:** LINK by default. Each consuming skill (draft-spec, socratic-grill, write-plan) gets a Phase 0 line: `If methodology terminology in this spec/grill/plan feels ambiguous (e.g., 'slice', 'phase', 'dispatch group', 'HITL', 'AFK', 'pgroup'), Read 'docs/agents/GLOSSARY.md' immediately before proceeding.` This trusts the agent + saves ~2KB of always-loaded context per skill invocation. Revisit trigger: if dogfood evals show agents skipping the GLOSSARY read when needed (>1 in 5 runs), escalate to LOAD in v1.10.
- **Spec update:** Slice 5 acceptance criteria specify LINK form with the exact instruction text. Add a v1.10 revisit trigger to the Revisit-triggers section.

### Item 8 — mtime-check reference file location

- **Starting state:** Three options: `skills/prior-art-research/references/`, `docs/agents/references/`, `skills/_shared/references/`.
- **Axes grilled:** Migration (10 SKILL.md files must point to one place), Reversibility (file move is cheap but breaks references).
- **Key questions:** Whose responsibility is the file? Does any existing convention point one direction? Does adding `docs/agents/references/` create a new layout pattern?
- **Resolution:** DECIDED.
- **Decision:** `docs/agents/references/system-context-staleness-check.md`. Rationale: cross-cutting concern; not owned by any single skill; `docs/agents/` is already the canonical location for chain-shared artifacts (`adrs/`, `specs/`, `plans/`, `dispatches/`); `references/` is a natural extension. `skills/_shared/references/` would create a new sub-convention solely for this one file — overkill. `skills/prior-art-research/references/` implies ownership by that skill, which contradicts the DRY goal.
- **Spec update:** Slice 6 acceptance criteria pin the exact path. Add a note that this introduces `docs/agents/references/` as a new directory convention — future cross-cutting helpers (verification protocols, citation conventions, etc.) land here.

### Item 10 — mtime-check failure-mode fallback

- **Starting state:** Implicit. What if `git log --since` fails (shallow clone, missing manifests, non-git context)?
- **Axes grilled:** Failure modes (silent skip vs noisy proceed).
- **Key questions:** Should the chain HALT if mtime-check can't run, or PROCEED with a warning?
- **Resolution:** DECIDED.
- **Decision:** PROCEED with an advisory. If `git log --since` returns non-zero exit or empty output for unexpected reasons (shallow clone is the canonical case), the helper falls back to: "load SYSTEM_CONTEXT.md without staleness check; emit single advisory line `⚠ Could not verify SYSTEM_CONTEXT.md freshness (git history unavailable). Proceeding — context may be stale.`" The shared reference doc documents this explicitly. Same fallback if `docs/agents/SYSTEM_CONTEXT.md` doesn't exist (load fails silently → emit a single warning → continue with empty cache).
- **Spec update:** Slice 6 acceptance criteria gain: "Shared reference doc documents the fallback for git-history-unavailable and SYSTEM_CONTEXT-missing cases. Both fallbacks: advisory line, proceed."

### Item 11 — Version bump cadence

- **Starting state:** Slice 1 mentions "1.9.0-alpha or similar" — under-specified.
- **Axes grilled:** Migration (per-slice alphas mean 6 marketplace releases vs 1).
- **Key questions:** What's the user impact of per-slice releases? Does the marketplace surface alpha tags differently?
- **Resolution:** DECIDED.
- **Decision:** Single bump at end. Version stays `1.8.0` while slices 1-5 land as PRs on `main`. When slice 6 merges, a release commit bumps `package.json`/`.claude-plugin/marketplace.json`/`.claude-plugin/plugin.json` to `1.9.0` simultaneously, then `gh release create v1.9.0` per established practice (per `claude-mem:version-bump` workflow conventions). CHANGELOG entries land per-slice under an `## Unreleased` header during the slices; on release, that header gets renamed to `## v1.9.0 — 2026-MM-DD`.
- **Spec update:** Slice 1 acceptance criteria drop "version bumped to 1.9.0-alpha or similar" — replaced with "CHANGELOG.md gains entry under `## Unreleased`." All other slices follow the same pattern. New top-level Release section in the spec specifies the post-slice-6 version-bump procedure.

### Item 12 — Slice 2 chain-integrity acceptance test

- **Starting state:** Spec asked how to measure "every chain relationship survives" automatically vs manually.
- **Axes grilled:** Observability (how does CI know slice 2 is correct).
- **Key questions:** Where does the script live? What's its dependency surface? Does it run pre-commit or pre-merge?
- **Resolution:** DECIDED.
- **Decision:** Build a small bash/awk assertion script as part of `tests/dogfood/10-description-budget/` (the same dogfood scenario as slice 1). Script:
  1. Reads the pre-slice-2 commit (`git show HEAD~1:skills/<name>/SKILL.md` for each skill at the merge base of the slice-2 branch) and extracts each `next-skills: [a, b, c]` line.
  2. For each `(source_skill, target_skill)` pair extracted, greps the post-slice-2 source SKILL.md body for the target skill name appearing in: (a) a line beginning with `HANDOFF`, (b) under a `## See also` section heading, or (c) any prose line (case-insensitive match on the skill name).
  3. Fails (exit 1) if any pair has zero matches, with output listing the unmatched pair.
  - Runs as part of the slice-2 PR pre-merge dogfood suite. NOT a per-commit hook (too brittle for in-flight work).
- **Spec update:** Slice 2 acceptance criteria gain: "Chain-integrity script at `tests/dogfood/10-description-budget/chain-integrity.sh` passes on the slice-2 PR. Script logic documented in the test scenario's README."

## New decisions surfaced during grilling

1. **`docs/agents/references/` is now a directory convention.** Previously references lived only under each skill (`skills/<name>/references/`). The mtime-check helper introduces a chain-shared references location. Future cross-cutting protocols (citation conventions, common assertion scripts, shared verification rituals) should land here.
2. **Slice 3 acquires an `agent-factors-check` requirement.** This is a chain-self-application: the meta-skill that catches LLM-product gaps fires against habeebs-skill's own new LLM-product skill. Prevents `verify-output` from shipping with the gaps habeebs-skill claims to catch.
3. **CHANGELOG `## Unreleased` header convention.** Established here as the v1.9.0 release pattern; should be adopted permanently going forward (no per-slice version bumps).
4. **Three keystone skills get protected anti-trigger lists.** Slice 1's trim is uniform on capability-statement length but selective on anti-triggers — preserving precision where over-triggering would derail other work.
